# CLAUDE.md — School Operating System

## What This Is

Multi-tenant school management SaaS. Single PostgreSQL database, shared schema, Row-Level Security isolation. NestJS modular monolith backend, Next.js App Router frontend, BullMQ worker service. Bilingual English/Arabic with full RTL. Two initial school clients, architected to scale to hundreds.

## Phase Workflow — Three Sessions Per Phase

Each phase is built across three separate sessions. Planning and execution NEVER share a session — this reclaims the context window that planning chains consume.

### Session 1 — Plan (`/plan-phase P{N}`)

A dedicated session that reads the spec and produces a self-contained implementation plan file. No application code is written. The output is `plans/phases-plan/P{N}-plan.md`.

**After this session**: review the plan file. Catch spec misinterpretations before any code gets written.

### Session 2 — Execute (`/implement-phase P{N}`)

A fresh session that reads the plan file and builds everything it describes. It does NOT re-plan. The plan file is the primary instruction; the phase spec is a reference for validation.

**Output**: application code + `plans/phases-results/P{N}-results.md` + `plans/phases-testing-instruction/P{N}-testing.md`

### Session 3 — Test (`/test-phase P{N}`)

A fresh session that reads the testing instructions and writes/runs every test. Fixes bugs found during testing.

**Output**: test files + `plans/phases-testing-result/P{N}-testing-result.md`

### Why Three Sessions

Planning consumes ~50% of the context window with reasoning chains. By writing the plan to a file and starting fresh for execution, the execution agent gets the full context window for: the condensed plan (~10-15k tokens), the phase spec, relevant source code, and the code it's writing. This is the difference between the agent losing context mid-implementation and having room to work.

## Context Loading Strategy

As the codebase grows, the agent cannot load everything. Follow this guidance:

### Early Phases (P0–P3): Load freely
The codebase is small. The agent can read all prior results and most source files without context pressure.

### Mid Phases (P4a–P6b): Load selectively
- Always load: plan file, context.md, phase spec, CLAUDE.md
- Load prior results files (they're small summaries)
- Load only the specific source files the plan's Section 1 identifies as dependencies
- Do NOT load `plans/masterplan.md` — it's 5000+ lines and the plan file already extracts what's needed

### Late Phases (P7–P9): Load surgically
- Same as mid phases, but be even more selective with source files
- The plan file's Section 10 ("Key Context for Executor") carries forward the patterns the agent needs
- If the agent needs a pattern example, the plan should include the file path — the agent reads just that one file

Once execution begins (Session 2 or Session 3), you are authorised to act autonomously until delivery.

- Do not re-plan. The plan is approved. Execute it.
- Do not wait for human approval during execution.
- Treat all project-scope edit permissions as granted.
- Make any code, file, folder, config, dependency, refactor, or structural changes needed to implement the plan.
- Accept your own changes without asking for confirmation.
- Fix bugs, build failures, and implementation issues yourself.
- Continue until the phase is complete.

## Do Not Ask for Approval For
- starting execution
- applying edits
- accepting changes
- fixing bugs
- retrying after failures
- refactoring within scope
- implementation-level decisions
- file restructuring within scope

## Only Ask for Approval If
A material bug, blocker, or discovery forces a meaningful deviation from the plan, such as:
- scope changes
- architecture changes
- major strategy changes
- major unplanned functionality
- materially different implementation path

## Default Rule
If the work still aligns with the plan, continue autonomously.
If the work no longer aligns in a material way, stop and ask.

## Critical Rules — Read Every Time

### RLS — The #1 Rule

Every tenant-scoped table has `tenant_id UUID NOT NULL`. Row-Level Security is enforced at the database layer.

- Tenant context is set via `SET LOCAL app.current_tenant_id` at the start of every Prisma **interactive** transaction
- A Prisma middleware handles this — it reads tenant context from the request pipeline
- **Interactive transactions ONLY**: All tenant-scoped DB access MUST use `prisma.$transaction(async (tx) => { ... })`. The sequential/batch API `prisma.$transaction([...])` is PROHIBITED (PgBouncer transaction mode cannot guarantee connection affinity). Custom ESLint rule `no-sequential-transaction` enforces this.
- **NEVER write raw SQL outside the RLS middleware.** No `$executeRawUnsafe`, no `$queryRawUnsafe` anywhere else. A lint rule enforces this.
- Every BullMQ job payload MUST include `tenant_id`. The `TenantAwareJob` base class sets RLS context before any DB operation. Jobs without `tenant_id` are rejected at enqueue time.
- The `users` table is the ONE exception — it is platform-level, not tenant-scoped, no RLS. Guarded at the application layer.

### Regression Testing — Mandatory

Before considering any work complete, run the full existing test suite to verify nothing was broken by the changes. This is not optional.

- Run `turbo test` (or the relevant test commands for affected packages) after every feature addition, redesign, fix, or refactor
- If any existing test fails that was passing before your changes, you MUST fix the regression before proceeding
- This applies to all work — new features, bug fixes, redesigns, refactors, schema changes
- Do not skip this step. Do not treat it as a nice-to-have. A change that breaks existing functionality is not complete.

### No Drift

- Do not add tables, columns, endpoints, or features not specified in the phase file
- Do not "improve" the schema with your own ideas
- If something seems missing, flag it — don't invent it
- If a phase file says a table is built in a different phase, do not create it

### TypeScript Strict

- `strict: true` in all tsconfig files
- No `any` types. No `@ts-ignore`. No `as unknown as X` casting hacks.
- All API inputs validated with Zod schemas (defined in `packages/shared`)
- All JSONB fields have corresponding Zod schemas

### Naming Conventions

- Files: `kebab-case.ts` — e.g., `payroll-runs.controller.ts`
- Classes: `PascalCase` — e.g., `PayrollRunsService`
- Database columns: `snake_case` — e.g., `total_working_days`
- API endpoints: `kebab-case` — e.g., `/api/payroll-runs/:id/finalise`
- Zod schemas: `camelCase` with `Schema` suffix — e.g., `createPayrollRunSchema`
- Enums in Prisma: `PascalCase` values — but the PostgreSQL enum values are `snake_case` strings
- Use British English for user-facing terms: `finalise`, `organisation`, `colour` — but American English for code keywords: `color` in CSS variables

### API Patterns

- RESTful routes namespaced under `/api/`
- All mutations require authentication + permission check
- All tenant-scoped endpoints verify tenant context before processing
- Use `@RequiresPermission('payroll.finalise_run')` decorator on controllers
- Use `@ModuleEnabled('payroll')` guard to check tenant module toggle
- Return consistent error shapes: `{ error: { code: string, message: string, message_ar?: string, details?: Record<string, any> } }`
- Pagination: offset-based for lists, `?page=1&pageSize=20` (default 20, max 100). Response: `{ data: T[], meta?: { page: number, pageSize: number, total: number } }`
- All monetary values are `NUMERIC(12,2)` in the DB and `number` in API responses — never use floating point
- IDs are UUIDs everywhere

### Service Layer Rules

- Services contain business logic. Controllers are thin — validate input, call service, return response.
- Services throw typed exceptions (NestJS `HttpException` subclasses)
- Cross-module calls go through the service layer, never direct DB access across module boundaries
- Audit logging is handled by the `AuditLogInterceptor` on mutations — services don't manually log

### Prisma Patterns

- Schema lives in `packages/prisma/schema.prisma`
- Migrations are generated with `npx prisma migrate dev --name descriptive-name`
- Use `@default(dbgenerated("gen_random_uuid()"))` for UUID PKs
- Use `@updatedAt` for `updated_at` columns
- Use `CITEXT` for email fields (case-insensitive)
- Index names: `idx_{table}_{columns}` — e.g., `idx_payroll_runs_tenant_status`
- Soft deletes are status-based (e.g., `archived`), never a `deleted_at` column

### Background Jobs (BullMQ — `apps/worker/`)

- Every job type has a dedicated processor class
- Every job payload includes `tenant_id` — enforced by `TenantAwareJob` base class
- Failed jobs retry with exponential backoff (configurable max attempts per queue)
- Dead-letter queue for jobs that exceed max retries
- Job names: `{domain}:{action}` — e.g., `payroll:generate-mass-payslips`, `notifications:dispatch`

### Styling Rules

- Tailwind CSS with logical utilities ONLY
- **NEVER use physical `left`/`right`** — use `start`/`end` equivalents: `ms-4` not `ml-4`, `pe-2` not `pr-2`, `text-start` not `text-left`, `rounded-s-lg` not `rounded-l-lg`
- A lint rule enforces this — physical directional classes are build errors
- shadcn/ui components from `packages/ui/`, built on Radix primitives
- No inline styles. No CSS modules. Tailwind only.
- Keep background transparent on components — let the shell set backgrounds

### i18n Rules

- next-intl handles locale resolution: user preference → tenant default → `en`
- Translation files in `messages/{locale}.json`
- Use `useTranslations()` hook in client components, `getTranslations()` in server components
- LTR enforcement on: email addresses, URLs, phone numbers, numeric inputs, enrolment IDs
- Western numerals (0-9) in both locales
- Gregorian calendar in both locales
- Platform admin is English-only — no translations needed for `(platform)/` routes

### Component Patterns

- Server components by default. Add `'use client'` only when you need interactivity.
- Data fetching in server components via service calls
- Client components for forms, interactive charts, real-time calculation previews
- Charts: use Recharts (already in the stack)
- Rich text: TipTap with BiDi support, `dir` attribute preserved per block
- Forms: controlled components with Zod validation (same schemas as backend where possible)

### State Management

- Server state: React Server Components + revalidation
- Client state: React `useState`/`useReducer` for local UI state
- No global state library. No Redux. No Zustand.
- Auth state: JWT in memory, refresh via httpOnly cookie

### Sub-agents

- you may use sub-agents to execute implementation steps within a phase
- hard rule: you may use sub-agents sonnet 4.6 or Opus 4.6 models only.
- never spawn any Haiku model sub-agents.

## Testing

### Required Test Coverage

- **Every API endpoint**: at least one happy-path and one permission-denied test
- **Every RLS-scoped query**: a cross-tenant leakage test (query as Tenant A, verify Tenant B data is invisible)
- **Every calculation**: unit tests for payroll formulas, grade computation, payment allocation, fee generation
- **Every state machine**: test all valid transitions AND verify blocked transitions throw
- **Every Puppeteer template**: snapshot test in both locales (en, ar)

### Testing Stack

- Backend: Jest + Supertest for API integration tests
- Frontend: Vitest + React Testing Library
- E2E: Playwright (Phase 9)
- PDF snapshots: Jest snapshot testing against rendered Puppeteer output

### Test File Location

- Co-located with source: `payroll.service.spec.ts` next to `payroll.service.ts`
- E2E tests: `apps/web/e2e/` and `apps/api/test/`

## Database Migrations

- All schema changes go through Prisma migrations
- Migration names are descriptive: `add-payroll-tables`, `extend-staff-profiles-bank-details`
- Never edit a migration that has been applied to staging or production
- Seed data: `packages/prisma/seed.ts` — includes system roles, permissions, platform-level notification templates

## Encryption Pattern

Sensitive fields (Stripe keys, bank details) use AES-256 encryption:
- Encryption key stored in AWS Secrets Manager
- `encryption_key_ref` column stores the Secrets Manager ARN
- Decrypt only in memory during use — never log, never return in API responses
- API responses show only last 4 characters (e.g., `****4821`)
- All access to encrypted fields is audit-logged

## Sequence Number Generation

Receipt numbers, invoice numbers, application numbers, payslip numbers all use the `tenant_sequences` table with row-level `SELECT ... FOR UPDATE` locking to prevent duplicates under concurrency. Format: `{prefix}-{YYYYMM}-{padded_sequence}`.
