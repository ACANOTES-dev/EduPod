# Risk Ledger

## AUD-001 — Control-plane RLS exception paths weaken tenant-isolation credibility

- Severity: High
- Confidence: High
- Domain: Security
- Affected modules: `tenant-resolution`, `permission-cache`, `rbac`, platform control-plane tables
- Why it matters: The application cannot convincingly claim PostgreSQL RLS is the sole hard tenant boundary while control-plane reads of FORCE-RLS tables happen outside tenant-bound transactions.
- Evidence summary: `tenant-resolution.middleware.ts` and `permission-cache.service.ts` explicitly document outside-RLS control-plane reads; archived phase docs record a prior cross-tenant leak under superuser/BYPASSRLS behavior.
- Fix direction: Use a non-bypass runtime role, isolate control-plane lookups onto a deliberate non-tenant path or dedicated lookup tables, and fail startup if the app role can bypass RLS.
- Suggested priority: Now

## AUD-002 — Notification dispatch path is not safe against duplicate dispatch

- Severity: High
- Confidence: High
- Domain: Reliability
- Affected modules: `apps/worker/src/processors/notifications`, `apps/worker/src/processors/communications`
- Why it matters: The same notification rows can be re-enqueued before a worker changes state, creating a direct duplicate-delivery path.
- Evidence summary: `dispatch-queued.processor.ts` comments that it is marking rows as processing but only clears `next_retry_at` while leaving `status: 'queued'`.
- Fix direction: Add an atomic claim or lease state and enforce compare-and-swap semantics before enqueue and/or send.
- Suggested priority: Now

## AUD-003 — Failed notifications have retry intent but no scheduled retry path

- Severity: High
- Confidence: High
- Domain: Reliability
- Affected modules: `dispatch-notifications`, `retry-failed`, `cron-scheduler`
- Why it matters: Transient provider failures can become operationally permanent even though the code records `next_retry_at`.
- Evidence summary: `dispatch-notifications.processor.ts` writes `status: 'failed'` plus `next_retry_at`; `retry-failed.processor.ts` exists; `cron-scheduler.service.ts` registers only `notifications:dispatch-queued`.
- Fix direction: Schedule the retry processor or merge its logic into the queued-dispatch poller, then add an end-to-end worker regression test.
- Suggested priority: Now

## AUD-004 — External notification sends happen inside a long-lived database transaction

- Severity: High
- Confidence: High
- Domain: Reliability
- Affected modules: `tenant-aware-job`, `dispatch-notifications`
- Why it matters: Provider calls cannot be rolled back, but the surrounding transaction can. That creates duplicate-send risk on retry and holds DB connections open during network latency.
- Evidence summary: `TenantAwareJob.execute()` wraps the full job body in a Prisma transaction, and `dispatch-notifications.processor.ts` performs Resend/Twilio calls inside `processJob()`.
- Fix direction: Split into short claim/update transactions around an out-of-transaction provider send, with idempotency protection and queue timeouts.
- Suggested priority: Now

## AUD-005 — Approval request creation is non-atomic and lacks one-open-request protection

- Severity: High
- Confidence: High
- Domain: Reliability
- Affected modules: `approvals`, `finance`, `payroll`, `admissions`
- Why it matters: Orphaned approval requests or duplicate open requests can desynchronize business state from approval state.
- Evidence summary: `checkAndCreateIfNeeded()` returns a request id; finance, payroll, and admissions then perform separate follow-up writes to move the entity into a pending-approval state.
- Fix direction: Make the approval request creation and entity state change one interactive transaction and add a uniqueness guard for one non-terminal request per entity/action/tenant.
- Suggested priority: Now

## AUD-006 — Production deploy is not gated on CI success

- Severity: High
- Confidence: High
- Domain: Ops
- Affected modules: GitHub Actions deploy pipeline
- Why it matters: In a no-staging setup, a push to `main` can begin deployment before lint, tests, or build have proven the revision safe.
- Evidence summary: `deploy.yml` triggers directly on push to `main`; `ci.yml` is a separate workflow with no repo-visible dependency from deploy to CI success.
- Fix direction: Deploy only from a `workflow_run` or release pipeline that requires successful CI.
- Suggested priority: Now

## AUD-007 — Production deploy is non-deterministic and uses a mutable server checkout

- Severity: High
- Confidence: High
- Domain: Ops
- Affected modules: GitHub Actions deploy pipeline
- Why it matters: The code reaching production may differ from the triggering revision, and the unfrozen install fallback weakens reproducibility further.
- Evidence summary: The deploy script performs `git pull origin main` on `/opt/edupod/app` and falls back from `pnpm install --frozen-lockfile` to `pnpm install`.
- Fix direction: Deploy a pinned SHA or immutable artifact, remove the unfrozen fallback, and use versioned release directories or equivalent.
- Suggested priority: Now

## AUD-008 — Direct foreign-table reads undermine modular boundaries

- Severity: High
- Confidence: High
- Domain: Architecture
- Affected modules: `behaviour`, `pastoral`, `gradebook`, `gdpr`, `staff-wellbeing`, other hotspot domains
- Why it matters: Schema changes have a much larger blast radius than the Nest import graph implies, which raises extension and refactor risk.
- Evidence summary: `danger-zones.md` and `module-blast-radius.md` explicitly warn about Prisma-direct cross-module reads, and sampled hotspot services confirm the pattern.
- Fix direction: Introduce owner-based query facades for the most widely shared tables and block new direct foreign-table reads outside approved facades.
- Suggested priority: Now

- Confidence: High

## AUD-010 — The default backend green bar overstates refactor safety

- Severity: High
- Confidence: High
- Domain: Tests
- Affected modules: backend test pipeline, especially finance and RLS-sensitive flows
- Why it matters: The default backend pass does not include integration, RLS, performance, or `apps/api/test` suites, so refactors can look safe while meaningful higher-level coverage was never run.
- Evidence summary: `apps/api/jest.config.js` excludes `.rls.spec.ts`, `.performance.spec.ts`, `.e2e-spec.ts`, and `<rootDir>/test/`; `jest.integration.config.js` restores them.
- Fix direction: Make the integration/RLS/e2e lane a required CI check and use it in pre-flight decisions for risky backend changes.
- Suggested priority: Now

## AUD-011 — Frontend regression coverage is too shallow for the product surface

- Severity: High
- Confidence: High
- Domain: Tests
- Affected modules: `apps/web`
- Why it matters: A 336-page bilingual application cannot rely mainly on screenshot smoke tests and mirrored logic specs if routing, forms, locale behavior, and real user actions matter.
- Evidence summary: Playwright is configured only for `./visual`; sampled specs are screenshot-based; frontend unit tests do not mount components and drift from live route/nav logic.
- Fix direction: Add behavioral Playwright journeys and component-level tests that mount shipped code for access control, navigation, notifications, search, and critical school workflows.
- Suggested priority: Now

## AUD-012 — Worker test coverage is selective across a broad critical surface

- Severity: High
- Confidence: High
- Domain: Tests
- Affected modules: `apps/worker`
- Why it matters: Important processor-heavy domains can regress without direct test protection.
- Evidence summary: 26 matching processor specs were found for 87 processors; entire processor groups such as `early-warning`, `imports`, `payroll`, and `regulatory` had no matching processor specs.

## AUD-013 — Worker health and observability are too shallow

- Severity: High
- Confidence: High
- Domain: Ops
- Affected modules: worker bootstrap, worker health, API health
- Why it matters: Queue-heavy failures can persist while web and API still look healthy enough to pass shallow smoke checks.
- Evidence summary: Worker health is a static `{ status: 'ok' }` response, no worker Sentry bootstrap was found, deploy smoke tests only web and API, and API health checks only one BullMQ queue directly.
- Fix direction: Add dependency-aware worker readiness/liveness, instrument the worker, and include worker/queue validation in deploy smoke checks.
- Suggested priority: Now

## AUD-014 — MFA TOTP secrets are stored plaintext in the platform `users` table

- Severity: Medium
- Confidence: High
- Domain: Security
- Affected modules: `auth`, `schema.prisma`
- Why it matters: Anyone with read access to `users` can clone a second factor; the table is platform-level and non-RLS.
- Evidence summary: `schema.prisma` stores `mfa_secret` as plaintext text, and `auth.service.ts` writes/verifies against that value directly.
- Fix direction: Encrypt MFA secrets at rest with the existing encryption facility or a dedicated KMS-backed wrapper.
- Suggested priority: Next

## AUD-015 — Raw-SQL governance is weaker than the written policy

- Severity: Medium
- Confidence: High
- Domain: Maintainability
- Affected modules: ESLint custom rules, raw-SQL-using modules
- Why it matters: The documented safety rail is broader than the rule that actually runs in CI, which allows architectural drift while still looking compliant.
- Evidence summary: `no-raw-sql-outside-rls.js` only flags `Unsafe` raw methods, while tagged raw SQL appears across app and worker code.
- Fix direction: Either narrow the policy text or enforce all raw SQL through an allowlist or wrapper with tenant-safety expectations.
- Suggested priority: Next
