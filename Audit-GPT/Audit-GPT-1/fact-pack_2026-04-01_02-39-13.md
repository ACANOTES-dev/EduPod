## 9. CI, Deploy, And Operational Inventory

### Facts

- CI workflow (`.github/workflows/ci.yml`):
  - triggers on push to `main` and pull requests to `main`
  - installs dependencies
  - generates Prisma client
  - runs lint
  - runs type-check
  - runs `pnpm turbo run test`
  - runs build
- Deploy workflow (`.github/workflows/deploy.yml`):
  - triggers on push to `main`
  - serialized by GitHub Actions `concurrency: production-deploy`
  - deploys over SSH
  - also takes a server-side lock via `flock /tmp/edupod-deploy.lock`
  - runs install, build, Prisma migrate deploy, post-migrate SQL via `docker exec psql`
  - restarts `api`, `web`, `worker` via PM2
  - runs smoke tests against web and API, failing the workflow on failure
- Local ops:
  - `docker-compose.yml` provisions `postgres`, `redis`, `meilisearch`
  - no PM2 ecosystem file is present in the repo
- Developer hygiene:
  - `.husky/pre-commit` runs `pnpm lint-staged`
- `.env.example` includes local secrets/config placeholders for:
  - JWT
  - search
  - encryption fallback
  - Stripe
  - Resend
  - Twilio
  - AWS
  - Sentry

### Strong Signals

- Production deploy safety is more deliberate than a naive “push and restart” workflow: there is both workflow-level serialization and server-side locking.
- Operational shape is still single-environment and server-centric; the deployment remains mutable-worktree based.
