# Challenge Pass

Audit timestamp: `2026-04-01_02-39-13`

## 1. Findings That Held Up Under Challenge

- Notification dispatch reliability defects held up. Direct source inspection confirmed three separate problems in the same flow: queued notifications are not actually claimed before enqueue, failed notifications require a separate retry processor, and that retry processor is not registered in the cron scheduler. Severity stayed `High`, confidence stayed `High`.
- Deploy safety findings held up. The repo-visible deploy workflow still triggers directly on push to `main`, performs `git pull origin main`, and falls back from frozen to unfrozen install. Severity stayed `High`, confidence stayed `High`.
- Approval workflow integrity held up. Direct inspection confirmed `checkAndCreateIfNeeded()` is called first and the entity status update happens in a separate write in finance, payroll, and admissions. Severity stayed `High`, confidence stayed `High`.
- Backend green-bar gap held up. The default `apps/api` Jest config excludes `.rls.spec.ts`, `.performance.spec.ts`, `.e2e-spec.ts`, and the `apps/api/test` directory, while a separate integration config reintroduces them. Severity stayed `High`, confidence stayed `High`.
- Boundary-leak and god-module findings held up. The fact pack, architecture docs, and sampled hotspot services all point in the same direction: direct foreign-table reads and oversized multi-responsibility services are live structural issues, not grep artefacts. Severity stayed `High`, confidence stayed `High`.
- Frontend/browser test-gap findings held up. The Playwright suite is explicitly visual-only, and sampled frontend unit tests mirror logic instead of mounting shipped components. Severity stayed `High`, confidence stayed `High`.

## 2. Findings That Weakened Under Challenge

- The original security concern can support a strong claim that control-plane paths weaken the RLS hard-boundary story, but not a categorical claim that production definitively runs with `BYPASSRLS`. Production database-role state was not directly observed. The final wording therefore avoids asserting a live production misconfiguration as fact.
- Health-check findings were narrowed. The code clearly under-reports worker and queue degradation, but it was overstated to say health endpoints always report healthy. The stronger, accurate claim is that they can still return HTTP 200 and/or a degraded status while important queue or worker failures persist.

## 3. Findings That Were Reframed

- The retry defect was reframed from "failed notifications never retry" to "a retry processor exists, but the scheduled path that should invoke it is missing from the cron scheduler." This is more precise and still materially risky.
- The deploy-safety finding was split conceptually into two related issues: `CI gating` and `release determinism`. They reinforce each other, but they are not identical defects.
- The RLS issue was reframed from "RLS is bypassed" to "control-plane services and archived evidence undermine the claim that database RLS is the sole hard backstop." That framing better matches the actual evidence.
- Raw-SQL concerns were kept as a governance and enforcement problem, not elevated to a direct exploit claim. The lint rule and policy text diverge; that is real, but it is not proof that every tagged raw query is unsafe.

## 4. Remaining Uncertainties

- Branch protection rules, required checks, and environment protection rules were not observable from repository files alone. They could mitigate some deploy risk, but the repo itself does not enforce them.
- The live production PostgreSQL role was not observed. The code and docs strongly suggest a control-plane exception problem, but runtime role configuration could be better or worse than implied here.
- Frontend Playwright and browser-based flows were not executed in this audit run. Frontend conclusions are based on static inspection plus CI/workflow evidence, not a passing or failing browser run.
- BullMQ jobs were not executed against live Redis/PostgreSQL. Reliability conclusions are based on code paths and test shape, not runtime backlog inspection.
- The Codex environment allowed only six active agents at once. The seventh subagent was launched immediately after two completed agents were closed, so Phase 2 was functionally complete but not executed as a single seven-agent parallel batch.

## 5. Adjustments Made To Severity, Confidence, Or Scoring

- No issue remained at `Critical`. The codebase has several `High` severity concerns, but the evidence did not justify labeling any single finding as a confirmed catastrophic active failure.
- The RLS/control-plane issue kept `High` severity but was framed as a credibility gap and architectural weakness rather than a verified present-tense production bypass.
- Operational readiness stayed at `6/10`, not lower, because CI, deploy serialization, API env validation, API health checks, and API-side observability are all real strengths even though worker visibility and release safety are weaker than needed.
- Security stayed below the "strong health" band because the control-plane/RLS exception paths, plaintext MFA secrets, and raw-SQL governance gap are material.
- No score below `4` or above `8` survived challenge. The evidence supports a mixed-health verdict, not either a collapse narrative or a strong-health narrative.
