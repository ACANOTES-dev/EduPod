# PHASE 3: Admissions

**Duration estimate**: 2 weeks
**Dependencies**: Phase 2 complete (student/parent/household models, year groups)
**Requires shared context**: `00-shared-context.md`

## SCOPE

This phase builds the configurable admissions system: form builder, public application page, application review workflow, approval-gated acceptance, and application-to-student conversion. After this phase, schools can accept online applications from parents and convert them into student records.

## DATA MODELS

### 3.4 Admissions

#### `admission_form_definitions`
| Column | Type | Constraints |
|--------|------|-------------|
| id | UUID | PK |
| tenant_id | UUID | FK → tenants, NOT NULL |
| name | VARCHAR(255) | NOT NULL |
| base_form_id | UUID | NULL, FK → admission_form_definitions (self-referencing) |
| version_number | INT | NOT NULL |
| status | ENUM('draft','published','archived') | NOT NULL |
| created_at | TIMESTAMPTZ | NOT NULL |
| updated_at | TIMESTAMPTZ | NOT NULL |

**Constraint**: `UNIQUE (tenant_id, base_form_id, version_number)`
**Constraint**: `UNIQUE (tenant_id, name) WHERE base_form_id IS NULL AND status != 'archived'` — partial unique for root (v1) forms only.

**Versioning rule**: Editing a published form creates a new version. The new version has `base_form_id` pointing to the original (v1) form definition. `base_form_id` is NULL for v1 forms. Existing applications retain reference to the old version.

**Constraint**: `UNIQUE (tenant_id, application_number)`

**`application_number` generation**: `{tenant_branding.receipt_prefix is NOT used here}` — uses sequence `tenant_sequences WHERE sequence_type = 'application'`: format `APP-{YYYY}-{padded_sequence}`.

**Duplicate detection**: On submission, system checks for existing applications with matching `student_first_name + student_last_name + date_of_birth` in the same tenant. Matches are flagged for review but not blocked.

## FUNCTIONAL REQUIREMENTS

### 4.5 Admissions

**4.5.1 Configurable Form Builder**
- Admin creates form with fields (all supported types)
- Set field properties: label, help text, required, visibility, searchable, reportable
- Simple conditional visibility (show field X when field Y = value Z)
- Platform-enforced required fields: The form builder warns if `date_of_birth` is not marked as required: "Date of birth is required for student conversion. Applications without it will need manual entry at conversion time." This is a warning, not a block.
- **Acceptance**: form preview matches builder configuration

**4.5.2 Form Versioning**
- Published form is live for applications
- Editing a published form creates new version
- Old version archived, existing applications preserved against old version
- **Acceptance**: historic applications always display correctly against their form version

**4.5.3 Public Admissions Page**
- Publicly accessible (no login required)
- Displays published form for the tenant
- Parent fills out form, creates draft application
- On submit: parent must register or log in, application linked to parent
- Rate limiting: 3 submissions per IP per tenant per hour (Redis key: `ratelimit:admissions:{tenant_id}:{ip}`, TTL = 1 hour). Honeypot field — submissions with honeypot filled are silently rejected.
- **Acceptance**: public page accessible, form submittable, application created

**4.5.4 Application Review**
- Staff views submitted applications
- Can change status: submitted → under_review → pending_acceptance_approval / rejected
- Internal notes (not visible to parents)
- **Acceptance**: staff can progress applications through workflow, notes preserved

**4.5.5 Approval-Gated Acceptance**
- If `tenant_settings.admissions.requireApprovalForAcceptance = true`:
  - Moving to `accepted` requires approval
  - Approval request created, routed to configured approver role
  - No student record created until approval is executed AND conversion action is triggered
- If false: staff can accept directly
- **Acceptance**: no student created before approval execution when approval is enabled

**4.5.6 Application-to-Student Conversion**
- The conversion is NOT a one-click action. It opens a pre-populated, editable conversion screen.
- Required fields that block conversion: `student_first_name`, `student_last_name`, `date_of_birth`, `year_group_id`, `parent1_first_name`, `parent1_last_name`.
- Recommended fields (warn if missing, don't block): `parent1_email`, `parent2_email`.
- Parent email is the sole matching key for existing parent records. If parent email matches existing unlinked parent → prompt admin to confirm link.
- Conversion allowed without parent email (parent marked communication-restricted).
- Household created with `needs_completion = true` until emergency contact added.
- **Conversion side-effects** (single transaction):
  - Student created (status: `active`, entry_date: today)
  - Parent/household created or linked
  - `student_parents` and `household_parents` junctions created
  - Meilisearch index updated
  - Audit log with all entity IDs
- **Acceptance**: converted student appears in student list, linked to correct household

**4.5.7 Duplicate Application Detection**
- On submission, check for matching first name + last name + DOB in same tenant
- Flag for review but do not block
- **Acceptance**: duplicates are warned, not blocked

## EDGE CASES

### 5.3 Admissions

| Edge Case | Handling |
|-----------|---------|
| Duplicate application (name + DOB match) | Warn and route for review, don't block |
| No eligible approver | Block acceptance submission with clear error message |
| Form edit doesn't corrupt historic submissions | Versioning: new version created, old preserved |
| Application conversion when household exists | Prompt admin to link to existing household or create new |
