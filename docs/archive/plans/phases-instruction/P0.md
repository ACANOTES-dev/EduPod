# PHASE 0: Foundations

**Duration estimate**: 2 weeks
**Dependencies**: None (starting point)
**Requires shared context**: `00-shared-context.md`

---

## SCOPE

This phase sets up all infrastructure, tooling, and foundational patterns that every subsequent phase depends on. There are no domain-specific data models or business logic. The output is a working monorepo with CI/CD, authentication scaffold, tenant resolution, RLS middleware, i18n foundation, and all service connections.

## IMPLEMENTATION NOTES

### Tenant Resolution Middleware

Refer to Section 2.6 in shared context. The middleware must:

1. Extract hostname from incoming request
2. Query `tenant_domains` WHERE `domain = hostname` AND `verification_status = 'verified'`
3. If no match → static 404 (never redirect, never leak tenant existence)
4. If match → load tenant, check status (active/suspended/archived)
5. Inject tenant context into request pipeline

### RLS Middleware

Refer to Section 2.5 in shared context. The Prisma middleware must:

1. Set `SET LOCAL app.current_tenant_id` at the start of each transaction
2. All raw SQL queries are prohibited except within this middleware
3. A linting rule flags any `$executeRawUnsafe` or `$queryRawUnsafe` usage outside this pattern

### BullMQ Job Pattern

Every job payload must include `tenant_id`. Create a `TenantAwareJob` base class that sets RLS context before any DB operation. Jobs without `tenant_id` are rejected at enqueue time.

### Auth Scaffold

Refer to Section 2.7 in shared context. Implement:

- JWT (15-min expiry, in-memory) + refresh token (7-day, httpOnly cookie, Redis)
- Session metadata in Redis
- Password reset flow (email, 1-hour token, single-use, hashed)
- Brute force protection (progressive delay: 5 failures → 30s, 8 → 2min, 10 → 30min lockout)

### i18n Foundation

Refer to Section 2.9 in shared context. Set up:

- next-intl with `en` and `ar` locales
- Tailwind logical CSS enforcement (lint rule against physical left/right)
- HTML `lang` and `dir` attributes from effective locale
