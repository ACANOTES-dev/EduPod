# Code Review Checklists — Hotspot Modules

> Focus items for the four highest-risk modules. Run through the relevant section before approving any PR that touches these areas. These items are the non-obvious things that lint and type-check won't catch.

---

## Behaviour Module

### State Machine & Transitions

- [ ] Every status change goes through the `isValidTransition()` / `isValidSanctionTransition()` / etc. guard from `packages/shared/src/behaviour/state-machine*.ts` — no direct `status` field writes that bypass the map
- [ ] `converted_to_safeguarding` is projected to `closed` via `projectIncidentStatus()` at **every** response surface: API list, API detail, search index, PDF export, parent portal, entity history, hover cards (see DZ-13)
- [ ] The appeal `decide()` transaction touches sanctions, incidents, exclusion cases, amendment notices, and entity history — if the PR touches any of these, verify the transaction boundary still covers all six tables (see DZ-17)
- [ ] Document generation (`generateDocument()`, `autoGenerateDocument()`) does NOT run Puppeteer inside an RLS transaction — generation must stay async or the connection pool is at risk (see DZ-19)

### Participants & Constraints

- [ ] `removeParticipant()` checks remaining student count before DELETE — the `trg_prevent_last_student_participant` DB trigger is the safety net, but application-level check must remain (see DZ-15)
- [ ] Break-glass access grants have an expiry job registered; any new break-glass path uses per-tenant `jobId` dedup (`daily:behaviour:break-glass-expiry:{tenant_id}`)

### Notifications & Safeguarding

- [ ] Parent notifications are enqueued (not inline) on incident `draft -> active` when `follow_up_required` or category has `requires_parent_notification`
- [ ] When `parent_description` is missing on a negative incident with `severity >= send_gate`, the parent notification stays `pending` — verify there is a visible staff alert or a follow-up task created (see DZ-14)
- [ ] Legal holds are created on exclusion case creation and appeal submission for all linked entities; releasing the anchor hold alone does NOT release propagated holds unless `releaseLinked=true` (see DZ-18)

### Scope & Access

- [ ] Behaviour scope for `class`-scoped users resolves via `ClassStaff` → `ClassEnrolment`; if scope changes, verify class assignment and enrolment records are in sync (see DZ-16)
- [ ] Safeguarding `sealed` status is irreversible — any code path that can reach `resolved -> sealed` must confirm the caller holds `safeguarding.seal_access` permission before proceeding

## Pastoral Module

### Circular Dependency Guard

- [ ] No new constructor-injected imports cross the `PastoralModule` ↔ `ChildProtectionModule` boundary without `forwardRef()` — breaking this pattern causes a silent NestJS startup failure (see DZ-35)
- [ ] Any new service added to either module that needs the other module's service must use `@Optional() @Inject(forwardRef(() => XModule)) private readonly xService: XService`

### Concern Disclosure & Author Masking

- [ ] Sensitive concern records have author masking applied before being returned in list/detail responses — any new endpoint that lists or returns concern data must apply the same mask
- [ ] CP (child protection) disclosure rules are checked before sharing concern details outside the CP team — pastoral concerns that escalate to CP must not be visible to non-CP pastoral staff via the pastoral API
- [ ] `survey_responses` and `survey_participation_tokens` have no `tenant_id` and no RLS — all queries against these tables MUST join through `staff_surveys.tenant_id`; only `StaffWellbeingSurveyService` may access them directly (see DZ-27)

### SST & Referrals

- [ ] SST meeting `scheduled -> in_progress` enqueues `pastoral:precompute-agenda`; if meeting scheduling logic changes, verify this job is still enqueued
- [ ] Referral state transitions (`draft -> submitted -> acknowledged -> ...`) all use the `VALID_TRANSITIONS` map in `referral.service.ts` — no direct status field writes
- [ ] Pastoral intervention terminal transitions (`achieved`, `partially_achieved`, `not_achieved`, `escalated`, `withdrawn`) all require `outcome_notes` — validate this field is present before accepting the transition

### Escalation Chain

- [ ] Critical concern escalation re-enqueues itself outside the Prisma transaction (not inside) — if the processor crashes after commit but before re-enqueue, the daily `pastoral:overdue-actions` cron is the 24-hour backstop
- [ ] Early warning `compute-student` jobs are fire-and-forget side effects from `evaluate-policy`, `notify-concern`, and `attendance-pattern-detection` processors — if the EARLY_WARNING queue is down, the daily 01:00 UTC cron is the fallback; no immediate error is raised (see DZ-32)

## Auth Module

### Token Lifecycle

- [ ] Refresh tokens are rotated on every use — old refresh token must be invalidated in Redis before issuing a new one; confirm `deleteSession()` is called on the old session
- [ ] Access token expiry (`JWT_EXPIRY`) and refresh token expiry (`REFRESH_EXPIRY`) come from the shared constants, not hard-coded inline — verify any new token path uses the same constants
- [ ] The `mfa_token` (intermediate token issued after password check, before MFA verification) must be short-lived and single-use — it must not be reusable once the TOTP step is completed

### Permission Cache

- [ ] After any role permission change, membership status change, or role assignment change, `PermissionCacheService` must invalidate the cache for ALL affected users — not just the role being changed (see DZ-08)
- [ ] Stale elevated permissions are a security risk; stale reduced permissions lock out legitimate users — if a PR modifies permissions/roles/memberships, trace the invalidation call path end to end

### MFA & Break-Glass

- [ ] MFA verification is checked before any endpoint marked as sensitive — verify `@RequiresPermission` is not the only guard on sensitive mutation endpoints
- [ ] Break-glass access is time-limited and audit-logged; any new break-glass grant enqueues the expiry job with per-tenant dedup
- [ ] Password length is enforced at the Zod schema level (minimum 8 characters) — confirm any new password-setting endpoint uses the shared `changePasswordSchema` or equivalent

### Brute Force & Audit

- [ ] Brute force protection (`checkBruteForce`) is called before credential validation — not after — so blocked IPs/emails are rejected without touching the database
- [ ] Failed logins, lockouts, and suspicious activity are passed to `SecurityAuditService`; empty catch blocks that swallow auth errors are prohibited
- [ ] The `DpaAcceptedGuard` allowlist must be updated when adding any new onboarding or legal-remediation endpoint that must be reachable before DPA acceptance (see DZ-30)

## Finance Module

### Sequence Numbers & Atomicity

- [ ] Invoice numbers, receipt numbers, and payslip numbers are generated via `SequenceService.nextNumber()` with `SELECT ... FOR UPDATE` row-level locking — any new numbered entity must use the same service, not an ad-hoc sequence
- [ ] The `'refund'` sequence type is not in `SEQUENCE_TYPES` constant but works at runtime — if anyone adds validation against that constant, refund number generation breaks silently (see DZ-04)
- [ ] Payment allocation is atomic: `deriveInvoiceStatus()` recalculates invoice status inside the same interactive transaction as the payment write — these must not be split

### State Machine

- [ ] All user-initiated invoice transitions (`void`, `cancel`, `write-off`) go through `validateInvoiceTransition()` from `packages/shared/src/constants/invoice-status.ts` — direct status writes are prohibited
- [ ] System-driven transitions (`overdue` via cron, `issued` via approval callback, `partially_paid/paid` via `deriveInvoiceStatus`) are the only valid paths outside the invoice service — no new service should write invoice status directly
- [ ] "Can this invoice accept payments/credit notes/late fees/Stripe checkout" checks must use `isPayableStatus()` from the helper — do not re-implement the payable status list inline

### Monetary Precision

- [ ] All monetary values stored as `NUMERIC(12,2)` in the database — no `FLOAT` or `DOUBLE` in schema
- [ ] All monetary arithmetic in TypeScript uses integer-safe operations (multiply × 100 before rounding, divide × 100 after) or a decimal library — never JavaScript floating-point arithmetic directly on currency amounts
- [ ] API responses return monetary values as `number`, not `string` — downstream display components are responsible for formatting

### Approval & Worker Callbacks

- [ ] Payroll run `finalised` via approval callback happens in the worker, not the API — if the worker fails mid-payslip generation, some payslips may exist and others not; any change to this path must include idempotency checking
- [ ] Approval `callback_status` must be set to `executed` by the callback processor on success — if the processor throws after acting but before setting the status, the daily reconciliation cron will retry; ensure all callback processors are idempotent
- [ ] `pending_approval -> issued` for invoices is handled exclusively by `InvoiceApprovalCallbackProcessor` — the API `issue()` method only accepts `draft` status; do not add `pending_approval` to the user-initiated path
