## A. Facts -- Directly Observed Evidence

### CI Pipeline

1. **Four-job pipeline** (`ci`, `deploy`, `integration`, `visual`) in `.github/workflows/ci.yml`.
2. **Secret scanning** via `gitleaks/gitleaks-action@v2` runs on every push and PR.
3. **Dependency audit** (`pnpm audit --audit-level=high`) runs but with `continue-on-error: true` -- known transitive vulns (xlsx, glob, next, tar) are tracked for upgrade but do not block merges.
4. **15+ custom governance scripts** enforced in CI: RLS audit, raw SQL governance, module cohesion, module tiers, global providers, module boundaries, cross-module deps, worker processor specs, i18n check, hotspot budgets.
5. **Coverage gating**: unit tests with `test:coverage`, then a `check:test-gate` script verifying thresholds.
6. **Integration tests** run in TWO separate Postgres/Redis environments (ports 5432/6379 in `ci`, ports 5553/5554 in `integration`).
7. **Visual smoke tests** via Playwright with artifact upload on failure.
8. **Build step** runs at end of `ci` job -- build breakage is caught before deploy.
9. **Deploy job** uses `appleboy/ssh-action@v1.0.0` to SSH to production with 20-minute timeout. Deploy is gated behind `needs: [ci]` and a concurrency group with `cancel-in-progress: false`.
10. **No DAST (dynamic application security testing)** or dependency license scanning in CI.

### Deploy Script (`scripts/deploy-production.sh`)

11. **File lock** (`flock 9`) prevents concurrent deploys.
12. **Pre-deploy backup**: `pg_dump` in custom format to `/opt/edupod/backups/predeploy/` with 14-day retention.
13. **Preflight checks**: database connectivity (psql), Redis connectivity (redis-cli or Node fallback), Prisma migration status.
14. **Post-migrate verification**: `post-migrate-verify.sql` checks existence of critical tables (students, staff_profiles, invoices, notifications, tenant_sequences), verifies RLS is enabled, RLS policies exist, `set_updated_at` trigger function exists, and triggers on critical tables.
15. **Smoke tests** after deploy: web endpoint, API health, API readiness, worker health, and auth login endpoint (expects 400/401).
16. **Automatic rollback**: on smoke test failure, rolls back to previous SHA by rebuilding and restarting. Rollback guard prevents infinite recursion.
17. **Notifications**: Slack webhook and Telegram bot notifications on success, failure, and rollback.
18. **No blue-green or canary deployment** -- it is a single-server, in-place deploy with PM2 reload.
19. **Migration runs AFTER build** but BEFORE PM2 restart. The backup is taken AFTER build but BEFORE migration.
20. **No migration rollback automation** -- if `prisma migrate deploy` partially fails, there is no automatic revert of the migration. The pg_dump backup is the safety net.

### Environment Validation

21. **API**: Zod schema in `apps/api/src/modules/config/env.validation.ts` validates env BEFORE `NestFactory.create()`. Calls `process.exit(1)` on failure. Required vars: DATABASE_URL, REDIS_URL, JWT_SECRET (min 32 chars), JWT_REFRESH_SECRET (min 32 chars).
22. **Worker**: Separate Zod schema in `apps/worker/src/env.validation.ts` with same pre-bootstrap validation pattern. Required vars: DATABASE_URL, REDIS_URL.
23. **Deploy script**: `check_required_secrets()` validates DATABASE_URL, DATABASE_MIGRATE_URL, REDIS_URL, JWT_SECRET, JWT_REFRESH_SECRET, APP_URL before proceeding.
24. **ENCRYPTION_KEY is optional** in the API env schema. The `.env.example` documents it as required for "encrypting Stripe keys and staff bank details." This means the API can start without encryption capability for sensitive data.

### Observability

25. **Sentry integration** across all three services:
    - API: `@sentry/nestjs` with PII scrubbing (strips UUIDs from URLs, redacts PII keys from extras, deletes authorization/cookie headers). `sendDefaultPii: false`.
    - Worker: identical Sentry setup with `@sentry/nestjs`, PII scrubbing, `serverName: 'worker'`.
    - Web (frontend): `@sentry/nextjs` with server, edge, and client configs. Client includes session replay gated by cookie consent. Traces sample rate: 10%.
    - `SentryGlobalFilter` is registered as `APP_FILTER` in AppModule.
    - `@SentryExceptionCaptured()` decorator on `AllExceptionsFilter.catch()`.
26. **Structured JSON logging** in production via `StructuredLoggerService`. Each log line includes `timestamp`, `level`, `message`, `requestId`, `tenantId`, `userId`, `context`. Falls back to NestJS colored output in development.
27. **Correlation IDs**: `CorrelationMiddleware` generates UUID per request (or accepts `X-Request-Id` header), stored in `AsyncLocalStorage`, set on response headers.
28. **Request logging middleware** is development-only (skipped in production). Production relies on structured logger + Sentry.
29. **No log aggregation pipeline** (e.g., Loki, ELK) confirmed active. `.env.example` has `LOKI_PUSH_URL` placeholder but no corresponding ingestion code found.
30. **No Prometheus/metrics endpoint** -- health checks are HTTP-based but no `/metrics` for scraping.

### Health Checks

31. **API health module** with three endpoints:
    - `GET /api/health` -- checks PostgreSQL, Redis, Meilisearch, BullMQ (5 queues with per-queue alert thresholds), disk, PgBouncer stats, Redis memory. Returns healthy/degraded/unhealthy.
    - `GET /api/health/ready` -- same as full check (readiness probe).
    - `GET /api/health/live` -- always returns 200 if the process is alive (liveness probe).
32. **Admin health dashboard** at `GET /api/v1/admin/health` (requires AuthGuard + PlatformOwnerGuard). Aggregates API health + worker health check + delivery provider configuration status.
33. **Worker health module** at `GET /health` -- checks PostgreSQL, Redis, and 10 BullMQ queues with stuck-job detection (5-minute threshold).
34. **Health endpoints are excluded** from ThrottlerGuard via `@SkipThrottle()`.

### Monitoring Processors

35. **DLQ monitor processor**: scans failed-job counts across all queues every 15 minutes. Logs warnings and sends Sentry alerts on non-zero failed counts.
36. **Canary processor**: Three-phase design (ping, echo, check). Distributes echo jobs to critical queues, verifies they complete within SLA, alerts via Sentry on missed SLAs. Full liveness verification of the BullMQ processing pipeline.

### Rate Limiting & Throttling

37. **Global throttle**: `ThrottlerModule.forRoot([{ ttl: 60000, limit: 100 }])` -- 100 requests per minute per IP globally.
38. **Auth-specific rate limiting**: `RateLimitService` implements email-based brute force (progressive delays), IP-based login throttle, and account lockout (locks `locked_until` in database after threshold failures).
39. **Domain-specific rate limiting**: Separate `NotificationRateLimitService` and `AdmissionsRateLimitService` found.

### Graceful Shutdown

40. **API**: `app.enableShutdownHooks()` in main.ts. PrismaService implements `OnModuleDestroy` (disconnects). RedisService implements `OnModuleDestroy` (quits connection). PM2 `kill_timeout: 30000`.

42. **Meilisearch client** implements `OnModuleDestroy` per grep.

## C. Inferences -- Supported Judgements

1. **The deploy pipeline is production-credible for a single-server setup.** The combination of file-lock exclusion, pre-deploy backup, preflight checks, post-migrate verification, smoke tests, and automatic rollback is a mature deployment pipeline. It lacks the sophistication of blue-green or canary deployments, but those require multi-server infrastructure this project does not yet have.

2. **Migration failure is the highest-risk deploy scenario.** If `prisma migrate deploy` fails partway through (e.g., a migration applies but the next one errors), the system is in an inconsistent state. The pg_dump backup exists for manual recovery, but there is no automated migration rollback. This is acceptable given Prisma's design constraints (migrations are forward-only) but demands careful migration review.

3. **Log aggregation is the missing observability piece.** Structured JSON logging is in place and production-ready, but logs currently go to PM2's file rotation with no confirmed ingestion into a centralized system. The LOKI_PUSH_URL placeholder exists but nothing consumes it. This means incident investigation requires SSH to the server.

4. **Developer onboarding is above average.** The combination of setup.sh, GETTING-STARTED.md, docker-compose with health checks, .env.example with clear instructions, and BullBoard for queue inspection covers the major pain points. A new developer can be productive within an hour assuming Docker is running.

5. **The ENCRYPTION_KEY being optional is a design choice, not a bug.** It allows the API to start for tenants that have not yet configured Stripe or entered bank details. However, it means there is no startup guarantee that encryption is available, and a runtime failure will occur if encrypted fields are accessed without the key.

## D. Top Findings

### D1. No Centralized Log Aggregation

**Severity:** Medium
**Confidence:** High
**Why it matters:** Structured JSON logs are produced but only persist in PM2 log files on the single production server. Incident investigation requires SSH access. Log loss is possible on disk failure. No alerting on log patterns (e.g., error rate spikes) outside Sentry.
**Evidence:** `StructuredLoggerService` writes to stdout/stderr. PM2 captures these. `.env.example` has `LOKI_PUSH_URL` placeholder but grep for `loki` across TypeScript files returns zero ingestion code. `RequestLoggingMiddleware` is dev-only (explicitly skipped in production).
**Fix direction:** Ship structured logs to a log aggregation service (Loki, CloudWatch Logs, or Betterstack). A simple PM2 log shipper or a sidecar process that tails PM2 logs and forwards them would close this gap. Alternatively, add a pino transport or Winston transport that pushes directly.

### D2. ENCRYPTION_KEY Is Optional in Env Validation

**Severity:** Medium
**Confidence:** High
**Why it matters:** The API can start without encryption capability for Stripe keys and bank details. If an admin attempts to configure a payment gateway or view staff bank details, the operation will fail at runtime rather than at startup. In a multi-tenant SaaS handling financial data, silent degradation of encryption is a risk.
**Evidence:** `apps/api/src/modules/config/env.validation.ts` line 33: `ENCRYPTION_KEY: z.string().optional()`. `.env.example` lines 36-37 document it as "Used for encrypting Stripe keys and staff bank details."
**Fix direction:** Either make ENCRYPTION_KEY required in the Zod schema (safest), or add a startup warning log when it is missing, and add a guard in the encryption service that throws a clear error with remediation instructions.

### D3. No Automated Migration Rollback

**Severity:** Medium
**Confidence:** High
**Why it matters:** If `prisma migrate deploy` partially applies (e.g., 3 of 5 migrations succeed, 4th fails), the database is in a half-migrated state. The deploy script does not attempt to restore from the pg_dump backup automatically. The code rollback (to previous SHA) proceeds, but old code may be incompatible with the partially-migrated schema.
**Evidence:** `scripts/deploy-production.sh` lines 348-352: migration runs, then post-migrate SQL. The `rollback_release` function (line 292) rebuilds and restarts at the old SHA but does not restore the database. The pg_dump backup exists (line 163-168) for manual recovery.
**Fix direction:** Add a migration-failure handler in the deploy script that (1) detects migration failure exit code, (2) automatically restores from the pre-deploy pg_dump before proceeding with code rollback, (3) sends a critical notification. Alternatively, wrap each migration in a transaction if Prisma supports it for the operations involved.

### D4. No Production Request Logging

**Severity:** Low-Medium
**Confidence:** High
**Why it matters:** `RequestLoggingMiddleware` is explicitly disabled in production (line 14: `if (process.env.NODE_ENV !== 'development') return`). This means no HTTP access log exists -- there is no record of which endpoints were hit, response times, or status codes. Sentry captures errors but not normal request flow. The structured logger is used in application code but not for request-level telemetry.
**Evidence:** `apps/api/src/common/middleware/request-logging.middleware.ts` line 14.
**Fix direction:** Enable request logging in production with appropriate filtering (skip health checks, limit body logging). Write to structured JSON format matching the existing logger. Could be as simple as removing the NODE_ENV guard and adding a sampling rate.

### D5. `pnpm audit` Does Not Block CI

**Severity:** Low-Medium
**Confidence:** High
**Why it matters:** `pnpm audit --audit-level=high` has `continue-on-error: true`. Known high-severity vulnerabilities in transitive dependencies (xlsx, glob, next, tar) are documented but do not prevent deployment. A new high-severity vulnerability introduced via a dependency update would also be silently ignored.
**Evidence:** `.github/workflows/ci.yml` lines 55-56.
**Fix direction:** This is a known tradeoff (transitive deps are hard to fix). Consider adding an allowlist of known acceptable CVEs and failing on any new ones. The `pnpm audit --json` output can be parsed and diffed against a known-good baseline.

### D6. PM2 Cluster Mode with instances: 1

**Severity:** Low
**Confidence:** High
**Why it matters:** Both `api` and `web` are configured with `exec_mode: 'cluster'` but `instances: 1`. This means they have the overhead of cluster mode (IPC, master process) without the benefit of multiple workers. On a server with sufficient memory, increasing instances would improve availability during restarts and throughput.
**Evidence:** `ecosystem.config.cjs` lines 12-13 (api: cluster, instances 1) and lines 31-32 (web: cluster, instances 1).
**Fix direction:** This is likely an intentional single-server constraint. When the server has headroom, increase `instances` to 2 for zero-downtime restarts (`startOrGracefulReload` already handles this). Alternatively, set `instances: 'max'` with `max_memory_restart` as the constraint.

### D7. Health Check and Readiness Probe Are Identical

**Severity:** Low
**Confidence:** High
**Why it matters:** `check()` and `getReadiness()` in `HealthService` call the same `buildFullResult()`. A readiness probe typically checks only critical dependencies (DB, Redis) while a health check may include non-critical services. With the current design, a Meilisearch outage or high disk usage reports the readiness probe as "degraded," which could cause a load balancer to pull the instance from rotation unnecessarily.
**Evidence:** `apps/api/src/modules/health/health.service.ts` lines 257-263.
**Fix direction:** Make `getReadiness()` check only critical dependencies (PostgreSQL, Redis) and return healthy/unhealthy (no degraded). Keep the full check for the general health endpoint and admin dashboard.

## G. Scores

### Operational Readiness: 7.5 / 10

**Anchoring:**

- 10 = Blue-green deploys, centralized logging with alerting, distributed tracing, metrics dashboards, automated incident response, chaos engineering.
- 7-8 = Single-server but with strong safety nets (backup, rollback, smoke tests, health checks, Sentry), missing centralized logging and metrics.
- 5 = Basic CI/CD, no health checks, no rollback, no structured logging.

**Score justification:** The deploy pipeline is mature for a single-server setup. The four-layer safety net (preflight, post-migrate verify, smoke test, automatic rollback) is genuinely strong. Health checks are comprehensive. Sentry covers all three services with PII scrubbing. The missing pieces -- centralized log aggregation, production request logging, metrics/alerting pipeline, and automated migration rollback -- are what separate this from an 8-9. The ENCRYPTION_KEY optionality is a minor concern.

### Developer Experience: 8.5 / 10

**Anchoring:**

- 10 = One-command setup, instant feedback loops, comprehensive docs, perfect local-prod parity, automated everything.
- 8-9 = Quick setup (< 30 min), good docs, docker-compose matches prod, proper caching, git hooks enforce quality, Swagger available.
- 5 = Manual setup, sparse docs, inconsistent tooling.

**Score justification:** The setup.sh + docker-compose + .env.example + GETTING-STARTED.md combination is strong. Turbo caching, lint-staged, commitlint, pre-push architecture checks, and BullBoard create a cohesive developer experience. The docker-compose includes PgBouncer, matching production topology. The 15+ custom governance scripts enforced in CI catch architectural drift early. Minor deductions: no dev containers / devcontainer.json, the GETTING-STARTED.md could benefit from a "verify your setup is working" smoke test command, and the setup.sh is macOS-only.
