## B. Strong Signals

- The repo has real baseline discipline: install, generate, lint, type-check, test, and build are all wired into CI, and production deploys are serialized twice.
- API operational hardening is noticeably stronger than worker hardening: API has strict env validation, richer health checks, structured logging, and Sentry scrubbing; worker does not show the same depth.
- Deployment is still based on mutating a long-lived server checkout rather than promoting an immutable artifact or a pinned revision.
- Documentation and runtime expectations have drifted in more than one place, which suggests developer onboarding pain will recur rather than being a one-off typo.
- Frontend browser-level confidence is structurally lower than backend confidence: there is a dedicated Playwright setup, but the main CI path only executes package `test` scripts.

## C. Inferences

- Within repo automation alone, production can begin deploying before CI has proven the pushed commit is safe.
- The exact code and dependency graph reaching production are not guaranteed to match the triggering GitHub Actions revision.
- A developer who follows the documented local setup literally is likely to hit confusing partial failures: API/worker env loading will not line up with `.env.local`, and Meilisearch may silently fall back.
- Background-job failures are more likely to escape early detection than web/API failures because the worker is neither smoke-tested in deploy nor instrumented as deeply.

## D. Top Findings

### 1. Production deploy is not gated on CI success

- Severity: High
- Confidence: High
- Why it matters: A push to `main` can start a production rollout even if lint, type-check, tests, or build later fail. In a no-staging setup, that materially raises blast radius.
- Evidence: `deploy.yml` triggers directly on push to `main` (`.github/workflows/deploy.yml:3-6`). `ci.yml` is a separate workflow, also triggered on push/PR, with no repo-visible dependency from deploy to CI success (`.github/workflows/ci.yml:3-39`).
- Fix direction: Make deployment contingent on a successful CI result, for example by deploying from a `workflow_run` that requires CI success, or by introducing a release/promotion workflow that only runs on already-verified revisions.

### 2. Production deploys are non-deterministic and not pinned to the triggering revision

- Severity: High
- Confidence: High
- Why it matters: `git pull origin main` on the server means the release target is "whatever `main` is when the SSH script runs," not necessarily the triggering SHA. The fallback from frozen to unfrozen install further weakens reproducibility and incident recovery.
- Evidence: The deploy script uses `git pull origin main` (`.github/workflows/deploy.yml:35-36`) and `pnpm install --frozen-lockfile || pnpm install` (`.github/workflows/deploy.yml:38-39`) inside the long-lived `/opt/edupod/app` checkout (`.github/workflows/deploy.yml:33`).
- Fix direction: Deploy an exact commit SHA or an immutable build artifact, remove the unfrozen install fallback, and use versioned release directories or another immutable-release pattern instead of building in place.

### 3. Local environment guidance and runtime expectations are inconsistent

- Severity: Medium
- Confidence: High
- Why it matters: This creates avoidable onboarding friction and makes partial misconfiguration easy. The most likely outcome is "the app starts, but some capabilities quietly degrade," which is expensive for developers to debug.
- Evidence: Docs and setup point developers to `.env.local` (`README.md:41-47`, `scripts/setup.sh:133-136`, `.env.example:4-5`), while API and worker preload `.env` (`apps/api/src/instrument.ts:5-9`, `apps/worker/src/main.ts:5-7`). `.env.example` defines `MEILISEARCH_HOST` (`.env.example:26-29`), but runtime code expects `MEILISEARCH_URL` (`apps/api/src/modules/config/env.validation.ts:39-41`, `apps/api/src/modules/search/meilisearch.client.ts:17-22`, `apps/worker/src/base/search.helpers.ts` from search output). Production deploy additionally depends on `DATABASE_MIGRATE_URL` (`.github/workflows/deploy.yml:62-65`), which is not in `.env.example` and not validated in API env schema (`apps/api/src/modules/config/env.validation.ts:3-45`).
- Fix direction: Pick one env-file convention for API/worker docs and runtime, standardize on one Meilisearch variable name across docs and code, and document and validate `DATABASE_MIGRATE_URL` explicitly.

### 4. Worker readiness and observability are too shallow for a queue-heavy production system

- Severity: High
- Confidence: High
- Why it matters: This system depends heavily on background processing. If worker or queue infrastructure degrades, web and API can look healthy while critical asynchronous workflows silently fail or backlog.
- Evidence: The fact pack records 87 worker processors. `worker.module.ts` registers many queues and processors (`apps/worker/src/worker.module.ts:97-330`), but the worker health endpoint is static (`apps/worker/src/health/worker-health.controller.ts:3-8`), worker bootstrap has no pre-bootstrap env validation (`apps/worker/src/main.ts:13-24`), and no worker Sentry bootstrap was found. Deploy smoke tests only web and API (`.github/workflows/deploy.yml:83-113`). API health downgrades BullMQ failures to `degraded` rather than `unhealthy` (`apps/api/src/modules/health/health.service.ts:110-118`, `apps/api/src/modules/health/health.service.ts:168-179`), so queue trouble can still return HTTP 200.
- Fix direction: Add worker dependency-aware readiness/liveness checks, wire worker errors into the same observability stack as API/web, and include worker or queue smoke coverage in deploy validation.

### 5. Frontend browser-level regression coverage is missing from CI

- Severity: Medium
- Confidence: High
- Why it matters: This is a large bilingual/RTL Next.js app. Unit-level Jest alone is not enough to catch layout, routing, locale, and rendering regressions that only show up in the browser.
- Evidence: CI only runs `pnpm turbo run test` (`.github/workflows/ci.yml:35-36`). The web package test script is `jest --passWithNoTests` (`apps/web/package.json:5-12`), while a separate Playwright visual suite exists for LTR, RTL, and mobile (`apps/web/e2e/playwright.config.ts:3-67`).
- Fix direction: Add at least a slim Playwright job to CI for critical login/shell/RTL/mobile routes, and keep the broader visual suite either in CI or on a required nightly cadence with alerting.
