## 5. State Machines

### SchedulingRun Lifecycle

**States:** queued → running → {completed, failed} → {applied, discarded}

| Event                       | From State | To State                      | Side Effects                                                                                      |
| --------------------------- | ---------- | ----------------------------- | ------------------------------------------------------------------------------------------------- |
| Create (prerequisites met)  | (none)     | queued                        | enqueue to BullMQ 'scheduling' queue; config_snapshot assembled                                   |
| Worker starts processing    | queued     | running                       | solver_duration_ms tracking begins                                                                |
| Solve completes (success)   | running    | completed                     | result_json populated; proposed_adjustments assembled; soft_preference_score calculated           |
| Solve fails (timeout/error) | running    | failed                        | failure_reason set; diagnostics_refined_report computed                                           |
| User applies run            | completed  | applied                       | writes schedules to live timetable; applied_by_user_id, applied_at set                            |
| User discards run           | completed  | (deleted or marked discarded) | no timetable write; run remains queryable for audit                                               |
| User cancels queued         | queued     | failed                        | failure_reason: "Cancelled by user"; no worker processing (guard at solver-v2.processor.ts:97)    |
| User cancels running        | running    | failed                        | failure_reason: "Cancelled by user"; CP-SAT phase cannot be interrupted cooperatively (SCHED-027) |

**Constraints:**

- Only one queued/running run per academic_year_id per tenant
- Only completed/applied runs are adjustable (addAdjustment validation)
- Only completed runs are discardable/applicable

### TeacherAbsence Lifecycle

**States:** active (reported) → {cancelled, covered (via substitution)}

| Event                 | Field                              | Transition                 | Side Effects                                                                     |
| --------------------- | ---------------------------------- | -------------------------- | -------------------------------------------------------------------------------- |
| Admin reports absence | absence_type='admin_reported'      | (created)                  | reported_by_user_id, reported_at set; triggers cascade (if auto-cascade enabled) |
| Teacher self-reports  | absence_type='self_reported'       | (created)                  | reported_by_user_id = current user, reported_at set; runs cascade immediately    |
| Admin/teacher cancels | cancelled_at, cancelled_by_user_id | reported → cancelled       | cancellation_reason recorded; cascadeService.revokeOffersForAbsence() called     |
| Substitute assigned   | (via SubstitutionRecord)           | reported → (pending cover) | SubstitutionRecord created with status='assigned'; offer sent                    |

### SubstitutionOffer Cascade

**States (via SubstitutionOfferStatus):** pending → {accepted, declined, revoked}

| Event                          | From State | To State | Side Effects                                                                      |
| ------------------------------ | ---------- | -------- | --------------------------------------------------------------------------------- |
| Offer created (cascade)        | (none)     | pending  | staff notified; expiry timer set                                                  |
| Staff accepts                  | pending    | accepted | SubstitutionRecord.status='confirmed'; next offer tier stopped                    |
| Staff declines                 | pending    | declined | reason_declined recorded; escalates to next tier (if multi-tier) or back to admin |
| Admin revokes (cancel absence) | any        | revoked  | (all pending offers revoked)                                                      |
| Expiry (timeout)               | pending    | revoked  | auto-escalate if config allows                                                    |

### ExamSession Lifecycle

**States (ExamSessionStatus):** planning → published → completed

| Event               | From State | To State  | Side Effects                           |
| ------------------- | ---------- | --------- | -------------------------------------- |
| Create exam session | (none)     | planning  | draft mode; can add/edit slots         |
| Publish schedule    | planning   | published | schedule frozen; invigilators assigned |
| Complete session    | published  | completed | results finalized; cannot modify       |

**Parent-Facing Endpoints:**

**Student-Facing Endpoints:**

**Admin-Only Endpoints:**

- Run orchestration: schedule.run_auto, schedule.apply_auto, schedule.view_auto_reports

### Auto-Scheduling Run Lifecycle

**Workflow:**

1. User checks prerequisites: `POST /runs/prerequisites` → GET `prerequisite_checks: [{name, passed, reason}]`
2. (Optional) Run feasibility: `GET /runs/feasibility` → 10-point capacity sweep (informs user of 100% placement likelihood)
3. Trigger solver: `POST /runs/trigger` → returns run_id, queued status
4. Poll progress: `GET /runs/:id/progress` → phase (preparing/solving/complete/failed), elapsed_ms, etc.
5. Diagnostics: `GET /runs/:id/diagnostics` → categorised failure analysis with recommendations
6. (Optional) Simulate overrides: `POST /runs/:id/diagnostics/simulate` → project impact without modifying
7. Apply: `POST /runs/:id/apply` → writes schedules to live timetable, status → applied
8. (Or) Discard: `POST /runs/:id/discard` → run marked discarded, no timetable write

**Key Insights:**

- Prerequisites checked via `SchedulingPrerequisitesService.check()`: validates competencies assigned, requirements set, staff availability, no active run
- Feasibility is a **separate pre-solve check** (10 deterministic capacity checks) — not the full CP-SAT solver
- Diagnostics use i18n translator (diagnostics-i18n/) to categorise failures (e.g., "not enough maths teachers for Y9", "double-period conflicts")
- Simulation service allows what-if without data modification
- Run can be cancelled if queued/running (SCHED-027: running can't be interrupted mid-solve, just flagged)

### Substitution Cascade Lifecycle

**Workflow:**

1. Absence reported: `POST /absences` (admin) or `POST /absences/self-report` (teacher)
2. Cascade triggered: `SubstitutionCascadeService.runCascade()` (async or sync depending on config)
3. Offers created: SubstitutionOffer records created for eligible staff via tiers (primary tier → secondary tier, etc.)
4. Staff notified: notifications sent (email/SMS/in-app) with offer link
5. Accept: `POST /offers/:id/accept` → SubstitutionRecord created with status=confirmed; cascade halts
6. Decline: `POST /offers/:id/decline` → reason recorded; escalates to next tier or returns to admin
7. Expiry/Revoke: `POST /absences/:id/cancel` → revokeOffersForAbsence() revokes all pending offers

**Key Insights:**

- Two flows: admin-reported (`reportAbsence`) and teacher self-report (`selfReportAbsence`)
- Self-report can nominate a specific substitute (`nominated_substitute_staff_id`); if declined, escalates to admin
- Cascade is multi-tier: primary substitutes first, then secondary pool, then manual admin intervention
- Cover tracking records actual cover (per SubstitutionRecord) and fairness metrics (CoverTrackingService)
- AI ranking (`AiSubstitutionService.rankSubstitutes()`) factors competency, availability, and fairness to avoid over-loading one person

### Conflict Detection

**When triggered:**

- On schedule create/update: ConflictDetectionService.detectConflicts() called before write
- Returns array of conflicts (teacher double-booking, room conflict, class double-booking)
- API response includes `meta.conflicts` if any found (client shows warning; user can proceed or fix)

**Conflict types:**

- Teacher in two places at once (teacher_staff_id clash)
- Room in two places at once (room_id clash)
- Class in two places at once (class_id clash)

### Pinned Entries & Solver Respect

- Pin: `source=pinned`, `is_pinned=true`, optional `pin_reason`
- Solver assembly (SchedulerOrchestrationService.assembleSolverInputV3) reads pinned entries as fixed constraints — solver cannot move them
- Unpin allows solver to override in next run

### Exam Scheduling

**Flow:**

1. Create session: `POST /exam-sessions` (status=planning)
2. Add slots: `POST /exam-sessions/:id/slots` (multiple slots, one per exam)
3. Generate schedule: `POST /exam-sessions/:id/generate` (auto-assign rooms, times)
4. Assign invigilators: `POST /exam-sessions/:id/assign-invigilators` (create ExamInvigilation records)
5. Publish: `POST /exam-sessions/:id/publish` (status=published, freeze schedule)

**Key Insights:**

- Each ExamSlot represents one exam slot (subject, year group, date, time, room)
- Invigilators assigned to slots; exam_invigilation table tracks assignments
- Status lifecycle: planning → published → completed

### Scenarios (What-If Testing)

**Flow:**

1. Create scenario: `POST /scenarios` (stores config_snapshot, which is a solver input variant)
2. Run solver on scenario: `POST /scenarios/:id/solve` (scenario-specific constraints, no main timetable affected)
3. Compare scenarios: `POST /scenarios/compare` (side-by-side result analysis)

**Key Insights:**

- Scenarios isolated from main scheduling runs; no side effects to live timetable
- Used for testing hypothetical staffing changes, constraint adjustments, etc.
- result_json populated after solve; UI can show what-if outcomes

### Calendar Subscriptions (CalendarSubscriptionToken)

**Flow:**

1. Create token: `POST /calendar-tokens` (entity_type: teacher|class, entity_id)
2. Returns token (64-char hex)
3. Public endpoint: `GET /v1/calendar/:tenantId/:token.ics` (no auth required; token is the auth)
4. Returns iCalendar (.ics) format for webcal:// subscriptions
5. Revoke: `DELETE /calendar-tokens/:tokenId`

**Key Insights:**

- One token per entity (teacher or class)
- Token persists; can be revoked but not cycled (new token needed)
- Public endpoint is read-only, rate-limited by token
