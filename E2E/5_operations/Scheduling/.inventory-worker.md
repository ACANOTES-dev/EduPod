## 5. Auto-Scheduler Solver Run Lifecycle

1. POST /v1/scheduling-runs (RequiresPermission schedule.run_auto)
2. Check active run (status IN ['queued','running']) — line 53-66
3. Check prerequisites — line 70-77
4. Assemble solver input via SchedulerOrchestrationService.assembleSolverInputV3()
5. Create scheduling_runs row (status='queued', config_snapshot JSONB)
6. Enqueue 'scheduling:solve-v2' job
7. Worker dequeues, claims (queued→running)
8. Calls solveViaCpSatV3(SOLVER_PY_URL, timeout=max(120s, max_solver_duration+60s))
9. On completion: updateMany(where: { id, status: 'running' }) writes result_json
10. Statuses: queued→running→completed | running→failed (crash-retry) | queued→failed (enqueue error)

## 6. Substitution Cover Notifications

NOT a worker job. CoverTrackingService and CoverNotificationsService are API-layer; may enqueue NOTIFICATIONS queue jobs directly.

## 7. Dead Letter / Stuck Job Handling

- Lock: 5min, extended every 60s
- Stalled: maxStalledCount 2 → DLQ
- Stale-reaper: startup (30s grace) + cron (every 60s)

## 8. External Integrations

- Solver Sidecar HTTP: SOLVER_PY_URL (default http://localhost:5557), timeout 120-3660s
- No S3, no email, no webhooks from solver
