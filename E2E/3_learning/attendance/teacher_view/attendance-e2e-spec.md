# Attendance Module — Teacher E2E Test Specification

**Module:** Attendance (Sessions, Records, Daily Summaries — Teacher-scoped surface).
**Perspective:** Teacher — user with `attendance.view`, `attendance.take`. **NO** `attendance.take_any_class`, **NO** `attendance.manage`, **NO** `attendance.amend_historical`, **NO** `attendance.override_closure`, **NO** `attendance.view_pattern_reports`.
**Pages covered:** Admin hub `/en/attendance` reduced to the teacher's own sessions, plus `/en/attendance/mark/{sessionId}` for teacher-owned sessions only.
**Scrutiny:** **HIGHEST** in the attendance pack. Teacher is the user who runs this module every day, and the boundary between "my session" and "someone else's session" is the single most load-bearing security boundary in the module.
**Tester audience:** QC engineer OR headless Playwright agent. Assume zero prior project context.
**Last Updated:** 2026-04-18

## 1. Prerequisites & Multi-Tenant Test Data

### Tenant A — `nhqs`

Same tenant as admin spec §1. Additional teacher-specific seeding:

- **User:** `teacher@nhqs.test` / `Password123!` — Role = **Teacher**. Call this T1.
- **Staff profile:** T1's user is linked to a `StaffProfile` row (id = `T1_staff`).
- **Teaching allocations:** T1 owns ≥ 3 schedule rows across ≥ 2 classes on the test weekday — e.g. K1B 08:45-10:00 (Arabic), K2A 10:15-11:30 (English), K3A 13:00-14:15 (Math).
- **Own sessions (test date):** ≥ 6 sessions for today where `session.teacher_staff_id = T1_staff` — mix of `open` and `submitted` statuses. At least 1 that was auto-generated by cron with default_present records already seeded.
- **Other-teacher sessions:** ≥ 20 sessions for today where `teacher_staff_id` is some other staff profile (call them T2, T3, T4). These must NEVER appear in T1's list and must return 403 on direct API access for write.
- **Second teacher:** `teacher-other@nhqs.test` — Role = Teacher with their own staff profile. Used for the "try to mark someone else's session" test (§16).
- **Teacher without staff profile:** `teacher-orphan@nhqs.test` — has the teacher role + `attendance.take` permission but no `staff_profiles` row linked. Used for the NO_STAFF_PROFILE test (§17).

### Tenant B — `demo-b`

- Seed 1 teacher `teacher@demo-b.test` with ≥ 1 open session bound to that teacher. Capture `sessionB.id` + `recordB.id` for the hostile cross-tenant matrix (§26).

### Browser / device matrix

Desktop Chrome (latest stable) + 375px iPhone SE emulation.

## 16. NOT_SESSION_TEACHER Boundary — The Core Security Boundary

This is the single most important test in the module. T1 must NOT be able to mutate any session that is not theirs, regardless of attack vector.

| #     | Attack vector                                                                                                                            | Expected                                                                                                                                                                                                         | Pass/Fail |
| ----- | ---------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------- |
| 16.1  | Direct API `PUT /attendance-sessions/{T2_session_id}/records` as T1                                                                      | 403 `NOT_SESSION_TEACHER` with the documented message. DB state unchanged.                                                                                                                                       |           |
| 16.2  | Direct API `PATCH /attendance-sessions/{T2_session_id}/submit` as T1                                                                     | 403.                                                                                                                                                                                                             |           |
| 16.3  | T1 tries to `POST /attendance-sessions` with `{ class_id: T2_classId, date: today }` where the class's schedule for today is owned by T2 | Rejected. Either 403 because the service infers the teacher_staff from the schedule and T1 ≠ T2, OR the session is created successfully but with `teacher_staff_id=T2` — verify actual and document. Prefer 403. |           |
| 16.4  | T1 submits records on a default-present session that T2 was auto-assigned to (cron)                                                      | 403.                                                                                                                                                                                                             |           |
| 16.5  | T1 opens `/en/attendance/mark/{T2_session_id}` in the UI                                                                                 | Page renders in read-only mode (teacher has `attendance.view` so GET returns 200). No Save / Submit buttons present; if the UI somehow surfaces a Save, the PUT returns 403.                                     |           |
| 16.6  | Modified frontend bundle forces a Save call                                                                                              | Backend still 403.                                                                                                                                                                                               |           |
| 16.7  | T1's session_id tampered: swap the last 8 chars of a T1 id with T2's                                                                     | Backend treats as unknown id → 404. No partial leak.                                                                                                                                                             |           |
| 16.8  | T2's session passed via query param in a bulk save                                                                                       | Endpoint doesn't accept bulk save across sessions. Any attempt returns 400.                                                                                                                                      |           |
| 16.9  | T1 tries to amend a T2 record                                                                                                            | 403 — teacher lacks `attendance.amend_historical` entirely (the endpoint gate returns 403 before service-level teacher scope check runs).                                                                        |           |
| 16.10 | Audit trail on a rejected attempt                                                                                                        | Rejection logged (structured error). Response body includes `code`: `NOT_SESSION_TEACHER`, `message` with clear remediation.                                                                                     |           |
| 16.11 | Performance of rejected attempts                                                                                                         | Reject latency < 200ms (service short-circuits before record-write loop).                                                                                                                                        |           |

## 35. Observations from Walkthrough

Observed issues go here. Seed list of watchpoints:

- **O-TV1 (P2 candidate)**: Teacher can read `daily_attendance_summaries` for any student in the tenant, not just those in their classes. Consider scoping via class-staff-join.
- **O-TV2 (P2)**: `/en/attendance/upload` link might still render for teachers even though the API returns 403. Either gate the link on `attendance.manage` OR present a soft gate on page load.
- **O-TV3 (P2)**: `/en/attendance/exceptions` / `/en/attendance/scan` similarly — verify the sidebar/menu presentation.
- **O-TV4 (P3)**: If T1 deep-links to T2's session, the mark page renders in read-only mode. Consider adding a small "You are viewing another teacher's session" banner for clarity.
- **O-TV5 (P3)**: Audit trail on rejected NOT_SESSION_TEACHER attempts — worth logging as security event (for anomaly detection).
