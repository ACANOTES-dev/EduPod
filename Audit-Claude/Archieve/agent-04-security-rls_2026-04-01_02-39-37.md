# Agent 04: Security & RLS Audit

**Audit timestamp**: 2026-04-01  
**Agent**: 4 of N -- Security & RLS  
**Codebase**: School Operating System (multi-tenant SaaS handling children's data)

---

## A. Facts

### RLS Infrastructure

1. **264 Prisma models total**; 251 have `tenant_id`; 13 do not (all justified -- see below).
2. **All 251 tenant-scoped tables have RLS policies applied at the database layer.** The initial gap analysis appeared to show 4 tables missing from `policies.sql` and `post_migrate.sql`, but deeper inspection revealed the RLS statements were placed inline in their respective migration files:
   - `ai_processing_logs` -- RLS in `20260329120000_add_ai_processing_logs/migration.sql` (ENABLE + FORCE + policy)
   - `retention_policies` -- RLS in `20260329120000_add_retention_policy_tables/migration.sql` (ENABLE + FORCE + nullable tenant_id policy)
   - `retention_holds` -- RLS in `20260329120000_add_retention_policy_tables/migration.sql` (ENABLE + FORCE + policy)
   - `attendance_pattern_alerts` -- RLS in `20260324020000_attendance_default_present/migration.sql` (ENABLE + policy -- **FORCE is missing**)

3. **13 models without tenant_id** (all intentional):
   - Platform-level: `users`, `tenants`, `permissions`, `dpa_versions`, `security_incidents`, `security_incident_events`, `sub_processor_register_entries`, `sub_processor_register_versions`, `gdpr_export_policies`
   - Auth tokens: `mfa_recovery_codes`, `password_reset_tokens`
   - Anonymity-by-design (DZ-27): `survey_responses`, `survey_participation_tokens`

### RLS Middleware (`createRlsClient`)

4. Uses `set_config('app.current_tenant_id', $1, true)` with **parameterised binding** (not string concatenation). SQL injection via tenant_id is blocked.
5. UUID format validated via regex before any SQL execution. Both `tenant_id` and `user_id` are validated.
6. Context is set via `SET LOCAL` (transaction-scoped), ensuring no connection bleed between requests.
7. The `as unknown as PrismaClient` cast is the ONE permitted type-safety escape, confined to the RLS middleware and service layer.

### Worker RLS (`TenantAwareJob`)

8. Uses `$executeRaw` tagged template literals (safe parameterised SQL) for `set_config`.
9. Validates `tenant_id` presence and UUID format before any DB operation.
10. Jobs without `tenant_id` are rejected at execution time with explicit error messages.
11. `SYSTEM_USER_SENTINEL` (all-zeros UUID) prevents system jobs from matching any CP access grant.

### Authentication

12. **JWT signing**: Separate secrets for access (`JWT_SECRET`, 32+ char minimum enforced by Zod env validation) and refresh (`JWT_REFRESH_SECRET`, 32+ char minimum).
13. **Access token expiry**: 15 minutes. **Refresh token expiry**: 7 days. Session stored in Redis with 7-day TTL.
14. **Token type enforcement**: AuthGuard explicitly checks `payload.type !== 'access'` and rejects refresh/mfa_pending tokens used as Bearer tokens.
15. **Cross-tenant token check**: AuthGuard verifies `JWT.tenant_id === request.tenantContext.tenant_id`. A token issued for Tenant A cannot be used against Tenant B's domain.
16. **MFA support**: TOTP-based with `otplib`. Recovery codes (10 per user) stored as SHA-256 hashes. MFA token is short-lived (5 minutes).
17. **Brute force protection**: Redis-backed counter per email with configurable thresholds and sliding window. Progressive lockout delays.
18. **Password hashing**: bcrypt with cost factor 12.
19. **Session management**: Redis-backed sessions with user indexing. Individual and bulk revocation supported. Sessions invalidated on password reset.
20. **Password reset**: Token hashed with SHA-256 before storage. 1-hour expiry. Max 3 active tokens per user. All sessions killed on reset. Does not leak user existence ("If email exists, reset link sent").

### RBAC

21. **PermissionGuard**: Redis-cached permission lookup per `membership_id`. Supports single permission or OR-logic (any of multiple permissions grants access).
22. **Cross-tenant check in PermissionGuard**: Verifies `JWT.tenant_id === request.tenantContext.tenant_id` independently of AuthGuard's check (defense-in-depth).
23. **Permission denied logging**: All denied attempts logged via `SecurityAuditService` with endpoint, IP, user agent, and reason.
24. **No `@Public` or `@SkipAuth` decorator exists in the codebase.** Unguarded controllers are intentional for public endpoints.

### Unguarded Controllers (9 total -- all verified intentional)

25. `health.controller.ts` -- Health check endpoint, no tenant data.
26. `public-admissions.controller.ts` -- Public admission form + application submission. Rate-limited by IP. Uses `@CurrentTenant()` for tenant context from hostname.
27. `public-website.controller.ts` -- Public website pages. Read-only.
28. `public-contact.controller.ts` -- Contact form submission. Rate-limited (5/IP/hour).
29. `public-sub-processors.controller.ts` -- GDPR sub-processor register. Read-only public data.
30. `stripe-webhook.controller.ts` -- Stripe signature verification (HMAC). Requires `tenant_id` in metadata.
31. `webhook.controller.ts` -- Resend webhook. Svix HMAC-SHA256 signature verification.
32. `unsubscribe.controller.ts` -- Email unsubscribe via signed token. No auth needed by design.
33. `scheduling-public.controller.ts` -- Calendar ICS feed via 64-char random hex token in URL.

### Security Hardening

34. **Helmet**: Enabled globally in `main.ts`.
35. **CORS**: Regex-anchored to `*.edupod.app` subdomains. Tested: rejects `evil.edupod.app.attacker.com` (suffix attack). Allows http (development) and https.
36. **Swagger**: Disabled in production (`NODE_ENV !== 'production'`).
37. **Sentry PII scrubbing**: Authorization and cookie headers stripped. UUIDs replaced with `:id`. PII keys (student, parent, staff, name, email, phone) redacted from extras. `sendDefaultPii: false`.
38. **Environment validation**: Zod schema validates all env vars at bootstrap, before NestFactory.create(). App refuses to start if misconfigured.

### Encryption

39. **AES-256-GCM** with 12-byte random IV and auth tag. Format: `{iv_hex}:{authTag_hex}:{ciphertext_hex}`.
40. **Key versioning**: Supports V1 through V100. Current version configurable. Legacy keyRef backward compatibility (`aws`, `local` map to V1).
41. **Key validation**: 32-byte (256-bit) enforced at service construction. App fails to start if key is wrong length.
42. **Masking**: `mask()` method returns only last 4 characters for API responses.

### GDPR Infrastructure

43. **DpaAcceptedGuard**: Global APP_GUARD. Blocks ALL tenant-scoped API traffic unless current DPA version is accepted. Exempt paths: `/auth`, `/legal`, `/public`, `/invitations/accept`.
44. **Consent service**: Uses RLS transactions for all consent operations. Validates subject existence before granting. Prevents duplicate active consents.
45. **GDPR token mapping** (DZ-28): `gdpr_anonymisation_tokens` table -- documented as critical. No API endpoint exposes it.

### Custom ESLint Security Rules

46. `no-raw-sql-outside-rls` -- Catches `$executeRawUnsafe`/`$queryRawUnsafe` outside the RLS middleware.
47. `no-sequential-transaction` -- Catches `prisma.$transaction([...])` (sequential API incompatible with PgBouncer).
48. `no-physical-css-direction` -- Catches LTR/RTL-unsafe CSS.

### Raw SQL Usage

49. **`$executeRawUnsafe`**: Only in `rls.middleware.ts` (parameterised) and `partition-maintenance.processor.ts` (eslint-disable with comment explaining safety: table names from hardcoded constants, not user input).
50. **`$queryRawUnsafe`**: Only in `partition-maintenance.processor.ts` (parameterised `$1` binding for table existence check).
51. **Worker partition maintenance**: Uses raw SQL for DDL (CREATE TABLE PARTITION OF). Table names sourced from `PARTITIONED_TABLES` constant array, not user input. Acceptable risk with eslint-disable.

### `any` Usage in Production Code

52. **15 occurrences in non-test files**. All are Prisma enum casts (`as any`) for string-to-enum conversion (e.g., `scope: dto.scope as any`). One `tx: any` parameter in `invitations.service.ts`. No `@ts-ignore` or `@ts-expect-error` anywhere.

## B. Strong Signals

1. **Defense-in-depth tenant isolation**: Cross-tenant checks happen at THREE independent layers: (a) RLS at the database, (b) AuthGuard JWT tenant_id vs hostname, (c) PermissionGuard JWT tenant_id vs hostname. An attacker would need to bypass all three simultaneously.

2. **Parameterised SQL everywhere**: The RLS middleware uses `set_config($1, ...)` with parameterised binding, not string interpolation. The worker uses Prisma's `$executeRaw` tagged template literals. SQL injection surface is minimal.

3. **No auth bypass mechanisms exist**: No `@Public()` decorator, no `@SkipAuth()` decorator, no global auth bypass flag. Public endpoints are explicitly separate controllers.

4. **Comprehensive security audit logging**: Login failures, brute force lockouts, permission denials, password resets, MFA setup/disable, session revocations -- all logged via `SecurityAuditService`.

5. **The codebase takes children's data seriously**: Safeguarding status projection (DZ-13), anonymity-by-design for survey responses (DZ-27), GDPR token isolation (DZ-28), consent withdrawal immediacy (DZ-29), CP access grants with sentinel blocking.

---

## C. Inferences

1. The RLS catalogue is split across three locations (policies.sql, post_migrate.sql files, and inline migration.sql files). This organizational inconsistency creates audit difficulty -- it is easy to conclude tables are unprotected when they are actually protected in a different file. This is a documentation/process issue, not a security issue.

2. The 15 `as any` casts for Prisma enum conversion suggest a recurring friction between Zod-validated strings and Prisma's generated enum types. This is a type-safety gap but not a security risk.

3. The `attendance_pattern_alerts` missing `FORCE ROW LEVEL SECURITY` is likely a copy-paste omission in that migration, since every other table follows the ENABLE + FORCE pattern. The practical risk depends on whether the application database user is also the table owner.

4. The DPA guard's test bypass (`NODE_ENV === 'test' || JEST_WORKER_ID`) is safe only if those env vars are never set in production. This is a standard pattern but worth a deployment verification.

## D. Top Findings

### Finding 1: `attendance_pattern_alerts` Missing `FORCE ROW LEVEL SECURITY`

**Severity**: Medium  
**Confidence**: High  
**Why it matters**: Without `FORCE ROW LEVEL SECURITY`, the table owner (typically the same database user Prisma connects as) bypasses RLS policies entirely. If the application connects as the table owner, ALL tenant data in `attendance_pattern_alerts` is accessible without tenant isolation. This table contains student attendance alert data -- sensitive information about children.  
**Evidence**: `packages/prisma/migrations/20260324020000_attendance_default_present/migration.sql` line 53: `ALTER TABLE attendance_pattern_alerts ENABLE ROW LEVEL SECURITY;` -- no `FORCE` statement follows. All 246 other RLS-protected tables have both `ENABLE` and `FORCE`.  
**Fix direction**: Add `ALTER TABLE attendance_pattern_alerts FORCE ROW LEVEL SECURITY;` via a new migration. Also add the table to `policies.sql` for canonical tracking. Also rename the policy from `tenant_isolation_policy` to `attendance_pattern_alerts_tenant_isolation` for consistency.

### Finding 2: RLS Policy Catalogue Fragmentation

**Severity**: Low  
**Confidence**: High  
**Why it matters**: RLS policies are spread across `policies.sql`, 26 `post_migrate.sql` files, and at least 3 inline `migration.sql` files. This makes comprehensive auditing error-prone. The initial gap analysis in the main audit session was "inconclusive" because of this fragmentation, and even this agent's systematic analysis initially showed 4 false positives before deep inspection.  
**Evidence**: `ai_processing_logs`, `retention_policies`, `retention_holds`, and `attendance_pattern_alerts` all have RLS defined inline in migration.sql rather than in `post_migrate.sql` or `policies.sql`.  
**Fix direction**: Add all 4 tables to `policies.sql` as the canonical reference. Establish a convention that `policies.sql` is the single source of truth for RLS catalogue, even if migration.sql files also apply the policies.

### Finding 3: Login Password Validation Minimum is 1 Character

**Severity**: Medium  
**Confidence**: High  
**Why it matters**: The login schema (`loginSchema` in `packages/shared/src/schemas/auth.schema.ts`) accepts passwords as short as 1 character: `password: z.string().min(1).max(128)`. While the registration schema correctly enforces `min(8)`, and the password reset schema enforces `min(8)`, the login endpoint itself would accept any string. This is defensively correct (you validate the hash, not the length, at login time), but the inconsistency means there is no server-side enforcement preventing users from having weak passwords if they are set through a path that bypasses the registration schema.  
**Evidence**: `packages/shared/src/schemas/auth.schema.ts` line 5: `password: z.string().min(1).max(128)` vs `user.schema.ts` line 5: `password: z.string().min(8).max(128)`.  
**Fix direction**: This is actually correct behavior -- login should accept any string and compare against the hash. The real question is: are there any password-setting code paths that do not enforce `min(8)`? Verify that all password mutation endpoints (registration, invitation acceptance, admin password set) use the `min(8)` schema.

### Finding 4: No Global API Rate Limiting

**Severity**: Medium  
**Confidence**: High  
**Why it matters**: While several specific endpoints have application-level rate limiting (admissions: IP-based, attendance scans: 50/tenant/day, contact forms: 5/IP/hour, notifications: per-parent per-channel), there is no global rate limiter (e.g., `@nestjs/throttler`) protecting the API as a whole. An attacker could make thousands of authenticated requests per second to any endpoint without hitting a rate limit.  
**Evidence**: No `ThrottlerModule` or `@Throttle()` decorator found in the codebase. Rate limiting is hand-rolled per-endpoint using Redis counters.  
**Fix direction**: Add `@nestjs/throttler` as a global guard with sensible defaults (e.g., 100 requests/minute per IP). Keep the existing per-endpoint rate limits for stricter control on sensitive paths.

### Finding 5: CORS Allows HTTP (Not Just HTTPS) Subdomains

**Severity**: Low  
**Confidence**: High  
**Why it matters**: The CORS regex `/^https?:\/\/[\w-]+\.edupod\.app(:\d+)?$/` allows `http://` origins alongside `https://`. In production with HTTPS enforced at the proxy layer, this is low risk, but it means a MITM on an HTTP connection could make cross-origin requests that the CORS policy would permit.  
**Evidence**: `apps/api/src/main.ts` line 37: regex pattern includes `https?`.  
**Fix direction**: If production is HTTPS-only (likely, given Cloudflare usage), tighten the regex to `^https:\/\/` only. Keep `http` in development by checking `NODE_ENV`.

### Finding 6: `survey_responses` Has No Database-Layer Tenant Isolation (DZ-27)

**Severity**: Medium (documented and mitigated, but inherently fragile)  
**Confidence**: High  
**Why it matters**: This is the only tenant-scoped data in the entire system with NO database-layer isolation. Tenant isolation depends entirely on application-layer query discipline (joining through `staff_surveys.tenant_id`). A single direct query against `survey_responses` without the join exposes all tenants' anonymous survey data. This is by design for anonymity, but the risk is real.  
**Evidence**: DZ-27 in `architecture/danger-zones.md`. The `SurveyResponse` model has no `tenant_id` column.  
**Fix direction**: Already documented. Ensure integration tests specifically verify no API path returns cross-tenant survey responses. Consider a database-level CHECK or trigger that prevents direct queries without a survey join.

### Finding 7: Partition Maintenance Uses String Interpolation in DDL

**Severity**: Low  
**Confidence**: High  
**Why it matters**: `partition-maintenance.processor.ts` line 125 uses string interpolation for partition names in DDL: `` `CREATE TABLE IF NOT EXISTS ${partitionName} PARTITION OF ${parentTable}` ``. While the values come from hardcoded constants (not user input), DDL cannot be parameterised with `$1` placeholders, and this is the only raw SQL construction in the codebase.  
**Evidence**: `apps/worker/src/processors/behaviour/partition-maintenance.processor.ts` lines 121-128. ESLint disable with safety comment present.  
**Fix direction**: Acceptable as-is. The table names derive from `PARTITIONED_TABLES` constant array. Add a validation assertion that partition names match `/^[a-z_0-9]+$/` before use.

### Finding 8: Invitations Service Has `tx: any` Parameter

**Severity**: Low  
**Confidence**: High  
**Why it matters**: `apps/api/src/modules/rbac/invitations.service.ts` line 389 has `tx: any`. This is the only untyped transaction client in the production backend. While the method is internal, the `any` type disables all type checking on the transaction, which could mask bugs where the wrong model is queried.  
**Evidence**: `invitations.service.ts:389: tx: any,`  
**Fix direction**: Type the parameter as `PrismaService` or `Prisma.TransactionClient`.

## G. Score

### Security Score: 8.5 / 10

**Anchored rationale**:

This is a genuinely strong security posture for a multi-tenant SaaS handling children's data. The highlights:

- **RLS is comprehensive**: 251 out of 251 tenant-scoped tables have RLS policies. The one defect (missing FORCE on `attendance_pattern_alerts`) is real but isolated.
- **Defense-in-depth**: Three independent layers of tenant isolation (database RLS, AuthGuard JWT check, PermissionGuard cross-tenant check).
- **Auth is production-grade**: Separate JWT secrets, short access token expiry, MFA support with recovery codes, brute force protection, session management, comprehensive audit logging.
- **No raw SQL leakage**: Custom ESLint rules enforce parameterised SQL. Only 2 files use raw SQL, both justified and documented.
- **GDPR infrastructure is mature**: DPA guard, consent management, GDPR tokenisation, privacy notices, sub-processor register.
- **The team documents danger zones**: 36 documented DZs including the 5 security-critical ones (DZ-13, DZ-27, DZ-28, DZ-29, DZ-30).

**What prevents a 9 or 10**:

- The missing `FORCE ROW LEVEL SECURITY` on `attendance_pattern_alerts` is a real defect affecting children's attendance data.
- No global API rate limiting exposes the system to brute-force attacks on non-protected endpoints.
- The RLS catalogue fragmentation makes auditing harder than it should be.
- The `survey_responses` table's application-only isolation is inherently fragile for a system handling children's data (though mitigated by design documentation and the specific anonymity requirement).

## H. Confidence in This Review

**High (85-90%)**

This review is based on direct source code inspection of all critical security files, systematic comparison of 251 Prisma models against RLS policies across all SQL files, and verification of all 9 unguarded controllers. The RLS gap analysis was performed programmatically (not sampling) and cross-verified against three sources (policies.sql, post_migrate.sql, migration.sql).

**Limitations**:

- I did not verify that the migrations have actually been applied to the production database (they exist in the repo, but deployment state is not verifiable from code alone).
- I did not test the actual HTTP endpoints for auth bypass -- this was a static analysis.
- I did not review every one of the ~130 controller files for endpoint-level security issues; I focused on the guard infrastructure and unguarded controllers.
- The `attendance_pattern_alerts` FORCE issue's real-world impact depends on whether the Prisma database user is the table owner, which requires database-level inspection.
