## B. Strong Signals (High-Confidence Observations)

1. **CI and deploy are decoupled** -- deploy triggers on push to main independently of CI. A force-push or merge that bypasses CI will deploy untested code to production.

2. **No automated rollback on deploy failure** -- smoke test failure exits non-zero (marks GitHub Action as failed) but the broken code remains deployed. Manual SSH intervention is required to rollback.

3. **Single mutable worktree deploy** (DZ-31) -- production builds occur in-place in `/opt/edupod/app`. The `concurrency: cancel-in-progress: true` setting means a second push cancels the first deploy mid-build, potentially leaving the server in a partially-built state with inconsistent artifacts.

4. **No pre-deploy database backup** -- migrations run during deploy without a snapshot first. If a migration is destructive and the deploy fails, there is no automated way to recover without the daily 3AM backup.

5. **Backups are on the same server** -- `/opt/edupod/backups/` is on the same disk as the database. A disk failure loses both the database and all backups.

6. **Fallback pnpm install in production** -- `pnpm install --frozen-lockfile || pnpm install` means a lockfile mismatch silently succeeds with a potentially different dependency tree in production. This is a supply chain risk.

7. **Worker lacks graceful shutdown hooks** -- API has `app.enableShutdownHooks()` but worker's `main.ts` does not. PM2 restart during deploy sends SIGTERM, which may terminate in-flight BullMQ jobs without completing them.

8. **No ecosystem.config.js in the repository** -- PM2 configuration is entirely server-side. There is no version-controlled definition of process names, instances, max memory, restart policies, or log rotation settings. Drift between documentation and actual PM2 config is undetectable.

9. **No integration or E2E tests in CI** -- CI runs only unit tests. The 19 Playwright e2e files and the integration test config (`jest.integration.config.js`) are not exercised in the pipeline.

10. **Structured logging but no aggregation** -- JSON structured logs are written to PM2 log files on disk. There is no log shipper, no centralized log platform, no ability to search or alert on log patterns across services.

## C. Inferences (Medium-Confidence)

1. **Deployment downtime is likely non-zero** -- the deploy script does `pm2 restart api web worker` which kills all three processes simultaneously. With a single instance of each, there is a window (likely 5-15 seconds) where the API, web, and worker are all down. The 5-second sleep before smoke test confirms this expectation.

2. **The backup drill script targets AWS RDS but the server is on Hetzner** -- `scripts/backup-drill.sh` uses `aws rds` commands extensively but the actual production database runs in Docker on Hetzner. This script appears to be pre-written for a planned migration to AWS or was copied from a template without being adapted. It would not work against the current production setup.

3. **Post-migrate scripts are applied every deploy** -- the deploy.yml loops over ALL `post_migrate.sql` files every deployment, not just new ones. While the SQL is idempotent (DROP IF EXISTS + CREATE), this adds latency and risk of accumulated scripts (currently 178 tables worth of RLS policies).

4. **No database connection pooling in the deploy workflow** -- migrations run via `DATABASE_MIGRATE_URL` (a separate connection string), suggesting awareness that the main URL goes through PgBouncer (which doesn't support DDL well). This is a positive signal.

5. **The `cancel-in-progress: true` on deploys is net-negative** -- while it prevents queueing, it means a rapid succession of pushes (e.g., a fix followed immediately by another fix) will cancel the first deploy mid-way, leaving the server in whatever state the cancelled build reached.

## D. Top Findings

### D-OPS-01: CI Does Not Gate Deployment (CRITICAL)

**What:** The deploy workflow (`deploy.yml`) triggers on `push: branches: [main]` independently of CI. CI and deploy run as separate, uncoordinated jobs. A merge to main that breaks CI will still deploy to production.

**Evidence:** Two separate workflow files with no `needs:` dependency or status check requirement. No branch protection rules requiring CI to pass before merge are enforced in the workflow definitions.

**Impact:** Broken code can reach production before CI reports failure.

**Recommendation:** Either make deploy a job in the CI workflow with `needs: [ci]`, or add a branch protection rule requiring CI to pass before merge to main.

### D-OPS-02: No Automated Rollback on Deploy Failure (HIGH)

**What:** When the post-deploy smoke test fails, the workflow exits with error status but the broken code remains live. There is no automated revert step.

**Evidence:** `deploy.yml` lines 112-114: if smoke test fails, `exit 1`. No preceding `git checkout HEAD~1` or PM2 restart with previous build.

**Impact:** Failed deploys leave the platform in a broken state until manual SSH intervention. The rollback procedure exists only in documentation (`MASTER-OPERATIONS-MANUAL.md` section 12).

**Recommendation:** Add an automated rollback block that triggers on smoke test failure: `git checkout HEAD~1 && pnpm build --force && pm2 restart all`.

### D-OPS-03: Backups Stored Only on Same Server (HIGH)

**What:** Database backups (`pg_dump` daily at 3AM) are stored at `/opt/edupod/backups/` on the same Hetzner server. No off-site replication.

**Evidence:** `MASTER-OPERATIONS-MANUAL.md` section 11: "Location: /opt/edupod/backups/", "Retention: 15 days". The backup drill script targets AWS RDS (not the actual Hetzner setup).

**Impact:** Disk failure, server compromise, or datacenter incident loses both the database and all backups simultaneously.

**Recommendation:** Add off-site backup replication (S3-compatible object storage on Hetzner, or a separate backup server). Update the drill script to match the actual infrastructure.

### D-OPS-04: Worker Lacks Graceful Shutdown (MEDIUM)

**What:** The worker service does not call `app.enableShutdownHooks()`. During PM2 restart (every deploy), SIGTERM may kill in-flight BullMQ jobs.

**Evidence:** `apps/worker/src/main.ts` -- no shutdown hooks. Compare with `apps/api/src/main.ts` line 63: `app.enableShutdownHooks()`.

**Impact:** In-flight background jobs (email dispatch, PDF generation, payroll calculations) may be interrupted during deployments, potentially leaving data in inconsistent states. BullMQ's retry mechanism provides some safety, but jobs that are partially complete may cause side effects on retry.

**Recommendation:** Add `app.enableShutdownHooks()` to the worker bootstrap. Consider implementing a drain period that waits for active jobs to complete before shutdown.

### D-OPS-05: No Centralized Log Aggregation (MEDIUM)

**What:** Structured JSON logs are written to PM2 log files on the server. There is no log shipping to a centralized platform.

**Evidence:** `StructuredLoggerService` writes to stdout/stderr. PM2 captures these to `/root/.pm2/logs/`. No log shipper configuration found in the codebase.

**Impact:** Debugging production issues requires SSH access and manual log file inspection. No ability to correlate logs across services, search historical logs, or set up log-based alerts. Log files are subject to PM2 log rotation and eventual deletion.

**Recommendation:** Add a log shipper (e.g., Vector, Promtail, or Filebeat) to forward JSON logs to a centralized platform (e.g., Grafana Loki, Elasticsearch, or Sentry log ingestion).

### D-OPS-06: Deploy cancel-in-progress Risks Partial Build State (MEDIUM)

**What:** `concurrency: { group: production-deploy, cancel-in-progress: true }` means a second push to main cancels any running deploy. The cancellation can occur at any point in the SSH script, leaving the server with partially built artifacts.

**Evidence:** `deploy.yml` lines 8-10. The server-side `flock` lock protects against parallel execution but does not protect against GitHub Actions killing the SSH session mid-execution.

**Impact:** Rapid pushes (common during hotfixes) can leave the server with mismatched build outputs (e.g., new API dist but old web .next).

**Recommendation:** Change to `cancel-in-progress: false` (queue deploys) or implement atomic deployment with symlink switching.

### D-OPS-07: PM2 Config Not Version-Controlled (LOW)

**What:** No `ecosystem.config.js` exists in the repository. PM2 process definitions (names, instances, memory limits, restart strategies) exist only on the server.

**Evidence:** No ecosystem config file found. Grep for `ecosystem` returned zero hits in source code.

**Impact:** PM2 configuration drift between documentation and reality. No way to reproduce the exact production process manager setup from the repository alone.

**Recommendation:** Add an `ecosystem.config.js` at the repo root defining all three processes with their production settings.

### D-DX-01: Excellent Local Development Setup (POSITIVE)

**What:** The local DX is well-designed: Docker Compose for infrastructure, setup.sh for prerequisites, seed-demo.sh for demo data, comprehensive .env.example, Turborepo for orchestration.

**Evidence:** `docker-compose.yml` with health checks for all services. `scripts/setup.sh` checking 6 prerequisites. `scripts/seed-demo.sh` with 2 tenant datasets and login credentials.

### D-DX-02: Comprehensive Operations Documentation (POSITIVE)

**What:** 2200+ line master operations manual covers every service, troubleshooting scenario, and emergency procedure. Separate runbook for quick reference. Pre-launch checklist with 200+ items.

**Evidence:** `Manuals/MASTER-OPERATIONS-MANUAL.md`, `Manuals/OPERATIONS-RUNBOOK.md`, `Manuals/PRE-LAUNCH-CHECKLIST.md`.

### D-DX-03: Strong Observability Foundations (POSITIVE)

**What:** Request correlation (AsyncLocalStorage), structured JSON logging in production with tenant/user context, comprehensive health checks with dependency latency, Sentry with PII scrubbing, UptimeRobot for external monitoring.

**Evidence:** CorrelationMiddleware, StructuredLoggerService, HealthService checking 5 dependencies, Sentry integration with `beforeSend` scrubbing on both backend and frontend.
