### A2. Custom ESLint Rules (3 rules in `packages/eslint-config/rules/`)

1. **`no-sequential-transaction`** -- Blocks `$transaction([...])` array form; enforces interactive transactions for PgBouncer compatibility
2. **`no-raw-sql-outside-rls`** -- Blocks `$executeRawUnsafe` / `$queryRawUnsafe` outside RLS middleware, tenant-aware base classes, migration files
3. **`no-physical-css-direction`** -- Blocks physical directional CSS (`ml-`, `mr-`, `pl-`, `pr-`, `left-`, `right-`, `text-left`, `text-right`, `rounded-l-`, `rounded-r-`, `border-l-`, `border-r-`) in favour of logical properties (`ms-`, `me-`, `ps-`, `pe-`, `start-`, `end-`, etc.)

All three rules are well-implemented, covering the three highest-risk convention violations.

### A3. ESLint Base Configuration

- `@typescript-eslint/no-explicit-any`: **error** (hard block)
- `@typescript-eslint/no-unused-vars`: error with `_` prefix exception
- `import/order`: warn (grouped, alphabetised, newlines between groups)
- `no-console`: warn (allows `warn` and `error`)
- Extends: `eslint:recommended`, `@typescript-eslint/recommended`, `prettier`

### A4. Empty Catch Blocks (`} catch {`)

- **Backend (`apps/api/src`)**: 30 occurrences across 20 files
- **Frontend (`apps/web/src`)**: 27+ occurrences across 20+ files
- **Worker (`apps/worker/src`)**: 2+ occurrences
- **Total across `apps/`**: ~621 occurrences of `} catch {` across 295 files

**Context**: Many are intentional fire-and-forget patterns (queue enqueue failures, notification dispatch), but the CLAUDE.md explicitly states: "Empty `catch {}` blocks are prohibited. Swallowing errors silently hides production bugs." The codebase violates this rule ~621 times. However, many of these are using the unnamed-catch pattern (`catch {` without a variable) which is a TypeScript 4.0+ feature meaning "I intentionally don't need the error object" -- this is different from `catch(e) {}` with an unused variable. Still, many lack even a `console.error` log.

### A5. Form Pattern Compliance

- **`react-hook-form` with `zodResolver`**: Found in only ~5 frontend files (regulatory/tusla, settings/legal, homework, early-warnings)
- **Individual `useState` per field**: Used in the vast majority of forms (including the reviewed `guardian-restrictions/page.tsx` with 7 individual `useState` calls for form fields)
- **CLAUDE.md rule**: "New forms must use `react-hook-form` with `zodResolver`... Individual `useState` per form field is not acceptable for new forms."
- **Conclusion**: Most forms use the legacy hand-rolled `useState` pattern. The `react-hook-form` adoption is in its early stages.

## B. Strong Signals (High-Confidence Inferences)

### B1. Convention Enforcement is Machine-Guarded Where It Matters Most

The three custom ESLint rules protect the three most critical architectural invariants: RLS transactions (PgBouncer safety), raw SQL containment (RLS bypass prevention), and RTL directionality (bilingual support). These are elevated to `error` level, making them hard gates. This is a strong positive signal -- the most dangerous violations cannot slip through CI.

### B2. Type Safety is Excellent

0 `@ts-ignore`, 0 `@ts-expect-error`, `no-explicit-any` set to `error`, only 15 `any` usages in production code across 412k LOC is an outstanding ratio (~0.004 per 100 LOC). The codebase is genuinely strict-mode TypeScript.

### B3. Empty Catch Blocks are the #1 Convention Violation

621 occurrences of `} catch {` across the codebase directly contradicts CLAUDE.md's "Empty `catch {}` blocks are prohibited" rule. While many are intentional (queue dispatch fallbacks, fire-and-forget operations), the lack of even a `console.error` means production bugs in these paths would be invisible.

### B4. react-hook-form Adoption is Near-Zero

Only ~5 files use `react-hook-form` with `zodResolver` out of hundreds of form-containing pages. The convention states "New forms must use react-hook-form" but virtually all forms use individual `useState`. This is a significant convention gap that impacts form validation consistency, error messaging, and accessibility.

### B5. i18n is Inconsistent Across Modules

The SEN reports page is fully internationalised. The guardian-restrictions page has extensive hardcoded English. This inconsistency suggests i18n was applied thoroughly in some modules but skipped or partially applied in others, particularly in the behaviour module's newer pages.

### B6. Large Files Follow Good Internal Structure

Despite having files over 1,000 LOC (15 backend files, 5 frontend files), the internal structure is clean: section separators, extracted private helpers, consistent naming. The size is driven by legitimate domain complexity (behaviour incidents, pastoral concerns, auth flows) rather than code smell. However, some extraction opportunities exist.

### B7. Code Duplication in Auth

`auth.service.ts` has two login paths (`login` and `loginWithRecoveryCode`) that share ~140 lines of near-identical credential checking, brute force handling, session creation, and token generation. This is a concrete DRY violation.

## C. Inferences (Lower Confidence)

### C1. Test Quality Appears Strong but Unevenly Distributed

571 test suites, 7,744 tests, all passing. The backend has 529 suites covering 56 modules. However, the frontend has only ~31 test files for 336 pages -- a ratio of ~1:11. Frontend testing is an area of significant underinvestment.

### C2. The `as unknown as PrismaService` Cast is Ubiquitous but Contained

Every RLS transaction uses this cast. It appears dozens of times across the codebase. While documented as the "one permitted exception," it does add cognitive overhead and makes type narrowing within transactions less trustworthy. A typed wrapper function could reduce this to a single point of type assertion.

### C3. Concern.service.ts Non-Atomic Share Operation May Be a Bug

`shareConcernWithParent` (line 751-804) reads the concern in one `$transaction` block, then updates it in a second `$transaction` block. Between the two transactions, the concern could be modified by another request (TOCTOU race). The read and write should be in a single transaction.

## D. Top Findings

### D1. [HIGH] Empty Catch Blocks Violate Stated Convention -- 621 Occurrences

- **Location**: Across 295 files in `apps/`
- **Impact**: Silent failures in production. Queue dispatch failures, notification errors, and data fetch failures would be invisible.
- **Convention**: CLAUDE.md states "Empty `catch {}` blocks are prohibited."
- **Recommendation**: Add at minimum `console.error` to every catch block. For user-triggered actions, add toast. Consider an ESLint rule (`no-empty-catch`) to enforce.

### D2. [HIGH] react-hook-form Convention Not Adopted

- **Location**: ~5 of hundreds of form-containing files use `react-hook-form`
- **Impact**: No standardised form validation, error display, or accessibility. Each form reinvents validation logic.
- **Convention**: CLAUDE.md states "New forms must use react-hook-form with zodResolver"
- **Recommendation**: Migrate forms to `react-hook-form` as they are touched. Prioritise forms with complex validation (create restriction, create incident).

### D3. [MEDIUM] i18n Gaps -- Hardcoded English in Newer Pages

- **Location**: `behaviour/guardian-restrictions/page.tsx` (column headers, sheet titles, form labels, constant maps)
- **Impact**: Arabic users see English strings for these elements. Breaks bilingual requirement.
- **Recommendation**: Extract all visible strings to translation keys. Audit all behaviour module pages for hardcoded English.

### D4. [MEDIUM] Auth Service Login Logic Duplication

- **Location**: `apps/api/src/modules/auth/auth.service.ts` -- `login()` (lines 195-451) and `loginWithRecoveryCode()` (lines 769-910)
- **Impact**: ~140 lines of duplicated credential check, brute force, session creation, token generation logic. Bug fixes in one path may not reach the other.
- **Recommendation**: Extract shared login flow into a private `_performLogin()` helper.

### D5. [MEDIUM] Non-Atomic Transaction in Pastoral Concern Sharing

- **Location**: `apps/api/src/modules/pastoral/services/concern.service.ts` -- `shareConcernWithParent()` (lines 751-804)
- **Impact**: TOCTOU race condition -- concern is loaded in one transaction, then updated in a separate transaction. Another request could modify the concern between reads.
- **Recommendation**: Merge read and update into a single `$transaction` block.

### D6. [LOW] Frontend Test Coverage Gap

- **Location**: `apps/web/src/` -- ~31 test files for 336 pages (~9% coverage by file count)
- **Impact**: UI regressions go undetected. The backend has ~529 test suites for production code.
- **Recommendation**: Prioritise integration tests for critical user flows (login, incident creation, finance).

### D7. [LOW] guardian-restrictions/page.tsx -- 19 useState Calls, No Form Library

- **Location**: `apps/web/src/app/[locale]/(school)/behaviour/guardian-restrictions/page.tsx`
- **Impact**: High state management complexity, manual validation, no error state per field
- **Recommendation**: Extract create/revoke forms into separate components. Adopt `react-hook-form` for the create form. Consider extracting detail sheet into `_components/`.

### D8. [INFO] `formatDate` Helper Duplication

- **Location**: Defined locally in `sen/reports/page.tsx` (line 191) but also imported from `@/lib/format-date` in `guardian-restrictions/page.tsx`
- **Impact**: Minor duplication; the local version may diverge from the shared one
- **Recommendation**: Use the shared `@/lib/format-date` consistently
