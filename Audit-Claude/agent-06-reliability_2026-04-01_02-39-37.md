# Agent 06: Reliability & Error Handling Audit

**Date**: 2026-04-01  
**Scope**: Failure handling, state machine integrity, job safety, retries, cron health, approval callback safety, dependency health checks  
**Codebase**: ~412k LOC production, 87 worker processors, 31 cron registrations, 20 queues, 36 documented danger zones

---

## A. Facts

### State Machine Enforcement

1. **24+ state machines documented** in `architecture/state-machines.md`, covering all major domain entities.
2. **Three enforcement patterns observed**:
   - **Pattern A (strongest)**: Shared `VALID_TRANSITIONS` map in `packages/shared` with `isValidTransition()` function, called from service methods. Used by: behaviour incident, sanction, intervention, exclusion, safeguarding, appeal, SEN support plans, SEN goals, invoice, student status, class enrolment, academic year/period.
   - **Pattern B (good)**: `VALID_TRANSITIONS` map local to the service file with a private `validateTransition()` method. Used by: form templates, contact form, critical incidents, SST meetings, referrals, referral recommendations.
   - **Pattern C (weak)**: "Implicit per-method" -- no transition map, status checks are scattered across individual methods with `if (run.status !== 'draft')` guards. Used by: payroll runs, payments, compliance requests. SEN referrals have **no enforced transitions at all** (doc states: "status is set directly via API update. All four values are valid enum values, transitions are not strictly enforced at service level").
3. **Engagement module** uses shared constants from `@school/shared` for event, submission, slot, and booking transitions -- consistent and well-placed.
4. **State machine documentation is current** (verified 2026-03-30), matches code evidence for the patterns I checked.

### TenantAwareJob Base Class

5. **75 lines**, clean implementation. Validates `tenant_id` presence and UUID format. Validates optional `user_id` format. Sets RLS context via safe tagged template literal (`$executeRaw` with parameterised values, not string interpolation). Defaults `user_id` to sentinel UUID for system jobs.
6. **Correlation ID logging** is present for cross-service tracing.
7. The `as unknown as PrismaClient` cast on line 70 is the documented permitted exception.

### Worker Job Retry Configuration

8. **All 20 queues have explicit retry config** in `worker.module.ts`:
   - Notifications: 5 attempts, 3s exponential backoff
   - Approvals, Scheduling, Security, Compliance: 2 attempts, 10s exponential backoff
   - All others: 3 attempts, 5s exponential backoff
9. **All queues have `removeOnComplete` and `removeOnFail`** configured, preventing unbounded Redis memory growth.

### Cron Scheduler

10. **31 cron registrations** in `CronSchedulerService` (646 LOC). Every cron job:
    - Uses `jobId: cron:${JOB_CONSTANT}` for BullMQ deduplication
    - Has `removeOnComplete` and `removeOnFail` retention limits
    - Runs in `OnModuleInit` lifecycle hook -- registered on worker startup
11. **Cross-tenant crons** (most of them) pass empty `{}` payload. The processor iterates all active tenants.
12. **Per-tenant dispatch crons** (behaviour daily/SLA/monthly) use an intermediate dispatcher pattern that iterates tenants and enqueues per-tenant jobs.

### Approval Callback Safety

13. **Approval callback chain is well-mitigated** (DZ-03 resolved):
    - `callback_status` field (`pending` / `executed` / `failed`) on `approval_requests`
    - `callback_attempts` counter
    - Error capture in `callback_error` field
    - Reconciliation cron at 04:30 UTC retries stuck callbacks
    - 5-attempt max before permanent failure (manual intervention required)
    - 30-minute stale threshold prevents racing with normal pipeline
    - Reconciliation caps at 100 requests per run to avoid overloading
14. **Three callback types mapped**: `announcement_publish`, `invoice_issue`, `payroll_finalise`. Adding a new type requires updating both the service map AND the reconciliation processor map -- documented as a danger in the event catalog.

### Health Checks

15. **API health service** checks 5 dependencies: PostgreSQL (latency), Redis (ping), Meilisearch (connectivity), BullMQ (stuck job count), Disk (free/total GB).
16. **Three endpoints**: `/health` (full check), `/health/ready` (readiness probe), `/health/live` (liveness -- always 200).
17. **Degraded vs unhealthy logic**: PostgreSQL or Redis down = `unhealthy` (503). Meilisearch, BullMQ, or disk down = `degraded` (200).
18. **Worker health is minimal**: Single endpoint returning `{ status: 'ok', service: 'worker' }`. No actual dependency checks.
19. **BullMQ health check only inspects the notifications queue** -- the other 19 queues are not checked.

### Error Handling Patterns

20. **79 empty `catch {}` blocks** across 49 API source files (non-test). **4 empty `catch {}` blocks** in worker source files.
21. **258 `catch (err)` blocks** across 114 API files that DO capture the error variable.
22. **Global exception filter** (`AllExceptionsFilter`) catches all unhandled exceptions, logs non-HttpException errors with stack trace, sends structured `{ error: { code, message } }` responses, and forwards to Sentry.
23. **Sentry integration**: API has full Sentry setup with PII scrubbing (UUID stripping, header removal, PII key redaction). Worker has **no Sentry integration** -- zero Sentry imports in `apps/worker/src`.
24. **NestJS typed exceptions** are widely used -- `NotFoundException`, `BadRequestException`, `ForbiddenException`, `ConflictException` found across the codebase. Raw `throw new Error()` found in 24 occurrences across 10 non-test module files -- mostly in infrastructure code (redis, s3, encryption) and communication providers, which is appropriate.

### Notification Dispatch Processor

25. **Full fallback chain implemented**: whatsapp -> sms -> email -> in_app.
26. **Per-notification retry with exponential backoff**: 60s \* 2^attempt, capped at `max_attempts` per row.
27. **Template resolution**: tenant-level first, then platform-level fallback.
28. **Separate retry-failed processor** scans for failed notifications eligible for retry and re-enqueues them.
29. **Template compilation cache** using SHA-256 hash keys -- prevents repeated Handlebars compilation.

### Evaluate Policy Processor

30. **Five-stage pipeline** (consequence, approval, notification, support, alerting) runs inside a single Prisma transaction.
31. **Idempotency**: Checks `evaluatedStages` before re-evaluating. Action execution has dedup guard checking for existing success records.
32. **Individual action failures are caught and logged** -- recorded as `execution_status = 'failed'` without aborting the pipeline.
33. **Cooldown mechanism**: 24-hour inter-escalation cooldown on rule+student+action combinations prevents duplicate actions.
34. **Early warning intraday trigger**: Exclusion-affected students are tracked and their risk profiles recomputed after the main transaction.

## B. Strong Signals

### Positive

1. **State machine culture is mature**: The codebase consistently uses transition maps with validation functions. The behaviour module (the most complex) has all state machines in `packages/shared` with dedicated test suites. The architecture doc accurately documents which service file guards each machine.

2. **Approval callback reconciliation is production-grade**: The DZ-03 mitigation is thorough -- tracked status, retry logic, permanent failure marking, stale threshold to prevent races, batch limits. This is one of the best patterns in the codebase.

3. **Worker retry/backoff is systematic**: Every queue has explicit configuration. The notification processor has an additional application-level retry with exponential backoff. The separation of BullMQ-level retries (queue config) from application-level retries (notification processor) is well thought out.

4. **Cron deduplication is universal**: Every cron registration uses `jobId` for BullMQ deduplication. This prevents duplicate cron executions across worker restarts.

5. **Global exception filter with Sentry**: All API errors are captured, structured, and forwarded to Sentry with PII scrubbing. The `AllExceptionsFilter` ensures no unhandled error returns raw stack traces to clients.

### Concerning

6. **79 empty catch blocks in API code is high**: While health checks (5), date parsing (2-3), and template rendering (1-2) account for some, the behaviour module alone has 4 empty catch blocks that silently swallow queue-add failures for parent notifications, policy evaluation, and auto-award checks.

7. **Worker has zero Sentry integration**: If a worker processor fails after exhausting retries, the error lands in Redis dead-letter queue and the BullMQ Dashboard (if deployed) -- but it is NOT reported to Sentry. This is a monitoring blind spot.

8. **Worker health check is a stub**: It returns `{ status: 'ok' }` unconditionally. A worker with a broken Redis connection or a dead Prisma client will still report healthy.

9. **BullMQ health check is narrow**: Only the notifications queue is inspected for stuck jobs. The behaviour queue (23+ job types, high-frequency SLA checks every 5 minutes) is not monitored via health.

## C. Inferences

1. **The empty catch blocks in `behaviour.service.ts` are intentional but risky**: The comments say "Don't fail the incident creation if queue add fails." This means parent notification, policy evaluation, and auto-award processing can silently fail with zero visibility. The incident is created but downstream automation never fires. Without Sentry in the worker and without logging in these catch blocks, there is no alerting mechanism.

2. **The "implicit per-method" state machines are technical debt**: PayrollRunStatus, PaymentStatus, and ComplianceRequestStatus lack explicit transition maps. While the current code may enforce transitions correctly through method-level guards, any future developer can add a new method that transitions without checking the valid paths. The pattern diverges from the rest of the codebase which uses explicit maps.

3. **SEN referral status has no enforcement at all**: The state machine doc explicitly notes transitions are "not strictly enforced at service level." This means the API can transition a referral from `report_received` back to `pending` -- a backward transition that should be impossible.

4. **The 30+ cron jobs create schedule congestion around 02:00-08:00 UTC**: Multiple heavy jobs run concurrently (gradebook risk detection at 02:00, MV refresh at 02:00 and 03:00, report card auto-generate at 03:00). DZ-06 and the event catalog both flag this, but no mitigation exists.

5. **Escalation self-chaining crash window (DZ-26, DZ-36) is a real risk**: Both safeguarding critical escalation and pastoral escalation timeout re-enqueue outside the Prisma transaction. If the worker crashes between commit and re-enqueue, the escalation chain silently terminates. The daily backstop cron provides recovery within 24 hours, but for critical safeguarding concerns, 24 hours of missed escalation is a significant gap.

## D. Top Findings

### Finding 1: Silent Queue-Add Failures in Behaviour Incident Creation

**Severity**: HIGH  
**Confidence**: 95%  
**Why it matters**: When a behaviour incident is created, three BullMQ jobs are enqueued: parent notification, policy evaluation, and auto-award check. All three queue-add calls are wrapped in empty `catch {}` blocks (`behaviour.service.ts` lines ~299, ~316, ~337, ~967). If Redis is temporarily down or the queue is misconfigured, these jobs silently fail. The incident is saved to the database, but:

- Parents are never notified about the incident
- Behaviour policy rules are never evaluated (no auto-escalation, no auto-sanctions)
- Awards for positive incidents are never checked

There is no log entry, no Sentry alert, and no mechanism for the user to know automation was skipped.

**Evidence**: `apps/api/src/modules/behaviour/behaviour.service.ts` -- four `catch { // Don't fail the incident creation if queue add fails }` blocks.

**Fix direction**: Replace empty catch blocks with `catch (err) { this.logger.error('[createIncident] Failed to enqueue behaviour:parent-notification', err); }` at minimum. Ideally, also set a flag on the incident (e.g., `automation_failed = true`) that surfaces in the UI so staff can manually trigger the automation.

---

### Finding 2: Worker Has No Sentry Integration

**Severity**: HIGH  
**Confidence**: 100%  
**Why it matters**: The API has full Sentry error tracking (`apps/api/src/instrument.ts` with PII scrubbing, `@SentryExceptionCaptured()` decorator on the global exception filter). The worker (`apps/worker/src/`) has zero Sentry imports or configuration. Worker processor failures after retry exhaustion go to the BullMQ dead-letter queue only. Without a monitoring dashboard watching Redis dead-letter queues in real time, production failures in the 87 worker processors are invisible to the engineering team.

**Evidence**: `grep "Sentry|@sentry" apps/worker/src/**/*.ts` returns zero matches. `apps/api/src/instrument.ts` has full Sentry setup. `apps/worker/src/` has no equivalent.

**Fix direction**: Add Sentry instrumentation to the worker app. Create `apps/worker/src/instrument.ts` mirroring the API's setup. Add `@SentryExceptionCaptured()` or a global error handler for BullMQ processor failures.

---

### Finding 3: Worker Health Check Is a No-Op Stub

**Severity**: MEDIUM  
**Confidence**: 100%  
**Why it matters**: The API health service checks PostgreSQL, Redis, Meilisearch, BullMQ, and disk. The worker health controller returns `{ status: 'ok', service: 'worker' }` unconditionally -- it does not check if the worker can reach Redis, if the Prisma client can connect to PostgreSQL, or if any queues are processing. If a process manager or load balancer uses this endpoint for health probes, a broken worker will still appear healthy.

**Evidence**: `apps/worker/src/health/worker-health.controller.ts` -- 9 lines, zero dependency checks.

**Fix direction**: Add at minimum Redis ping and Prisma `$queryRaw SELECT 1` checks. Optionally check that the worker's BullMQ connection is alive and that no queues have abnormally high waiting job counts.

---

### Finding 4: BullMQ Health Check Only Inspects One of 20 Queues

**Severity**: MEDIUM  
**Confidence**: 95%  
**Why it matters**: The API health service only inspects the `notifications` queue for stuck jobs. The other 19 queues (behaviour, pastoral, finance, compliance, etc.) are not monitored. A stuck or dead queue in the behaviour module (which runs SLA checks every 5 minutes for safeguarding concerns) would not be detected by the health endpoint.

**Evidence**: `apps/api/src/modules/health/health.service.ts` line 82 -- constructor injects only `@InjectQueue('notifications')`. The `checkBullMQ()` method only calls `this.notificationsQueue.getActive()`.

**Fix direction**: Inject all critical queues (or at least the top 5 by volume/importance: notifications, behaviour, pastoral, compliance, finance) and check each for stuck jobs.

---

### Finding 5: Three State Machines Lack Explicit Transition Maps

**Severity**: MEDIUM  
**Confidence**: 90%  
**Why it matters**: PayrollRunStatus, PaymentStatus, and ComplianceRequestStatus use "implicit per-method" guards instead of explicit `VALID_TRANSITIONS` maps. SEN referral status has no enforcement at all. This creates two risks: (1) A future method could transition to an invalid state without validation catching it. (2) The lack of a single source of truth makes it harder to reason about what transitions are valid. Every other major state machine in the codebase uses explicit maps -- these are exceptions.

**Evidence**: `architecture/state-machines.md` lines 239, 296, 517 document these as "(implicit)" or "(implicit per-method)". SEN referral: "transitions are not strictly enforced at service level." Grep for `VALID_TRANSITIONS` in finance/ and payroll/ directories returns zero matches for payroll, and only the invoice helper for finance.

**Fix direction**: Create `VALID_TRANSITIONS` maps for PayrollRunStatus and ComplianceRequestStatus in `packages/shared`. Add `validateTransition()` guards to the corresponding services. For SEN referrals, add at minimum a forward-only constraint.

---

### Finding 6: Escalation Chain Crash Window (Safeguarding + Pastoral)

**Severity**: MEDIUM  
**Confidence**: 85%  
**Why it matters**: Both `CriticalEscalationProcessor` (safeguarding) and `EscalationTimeoutProcessor` (pastoral) re-enqueue the next escalation step OUTSIDE the Prisma transaction. If the worker process crashes between transaction commit and BullMQ enqueue, the escalation chain silently terminates. For critical safeguarding concerns (DLP notification chain), this could mean up to 24 hours before the daily backstop cron detects the gap.

**Evidence**: DZ-26 and DZ-36 in `architecture/danger-zones.md` document this explicitly. The event-job-catalog confirms the re-enqueue is outside the transaction (line ~52 comment in the processor).

**Fix direction**: Two options: (1) Write the "next step needed" flag to the database inside the transaction, and have the backstop cron check for pending escalation steps more frequently (e.g., every 30 minutes instead of daily). (2) Accept the risk but reduce the backstop interval for critical/safeguarding concerns to 1 hour.

---

### Finding 7: No Dead-Letter Queue Monitoring UI

**Severity**: LOW  
**Confidence**: 100%  
**Why it matters**: Failed jobs go to dead-letter queues in Redis, but the event-job-catalog explicitly notes "there's no admin interface to inspect or replay them." Combined with the absence of Sentry in the worker, failed jobs after max retries are invisible unless someone manually checks Redis or runs BullMQ Dashboard. For compliance-sensitive operations (GDPR erasure, payroll finalisation), this is a gap.

**Evidence**: Event-job-catalog, Danger Zone #5 ("No dead-letter monitoring UI").

**Fix direction**: Deploy BullMQ Board or Bull Dashboard as a secured admin-only route. Or add a cron that counts dead-letter queue depth across all queues and alerts via Sentry/email when non-zero.

---

### Finding 8: Cron Schedule Congestion in 02:00-03:00 UTC Window

**Severity**: LOW  
**Confidence**: 80%  
**Why it matters**: At 02:00 UTC, three jobs run simultaneously: `gradebook:detect-risks`, `behaviour:refresh-mv-exposure-rates`, and the behaviour SLA check (every 5 min). At 03:00 UTC, two more: `report-cards:auto-generate` and `behaviour:refresh-mv-benchmarks`. These are all cross-tenant jobs that iterate every active tenant and perform heavy database operations (MV refresh, grade threshold checks, report card generation). Under multi-tenant load, concurrent PostgreSQL aggregations may cause memory pressure.

**Evidence**: `cron-scheduler.service.ts` lines 120-197. Event-job-catalog Danger Zone #9.

**Fix direction**: Stagger cron schedules by 30 minutes: MV refreshes at 01:30 and 02:30, gradebook detection at 02:00, report cards at 03:30.

## G. Score

### Reliability Score: 7/10

**Rationale**: The codebase demonstrates mature reliability engineering in several critical areas -- the approval callback reconciliation system is production-grade, state machine enforcement is consistent across the majority of domains, worker retry configuration is systematic, and cron deduplication is universal. The global exception filter with Sentry integration provides good API error visibility.

However, two significant gaps prevent a higher score: (1) the worker has zero Sentry integration, creating a blind spot for 87 processor failure modes, and (2) the 79 empty catch blocks in the API (particularly the 4 in `behaviour.service.ts` that silently swallow automation failures) violate the project's own "no silent failures" rule. The worker health check stub and the narrow BullMQ health monitoring compound the monitoring gap. The few state machines without explicit transition maps are minor but represent inconsistency in an otherwise disciplined codebase.

**What keeps it at 7 (not lower)**: The documented danger zones are accurate and honest. The cron system is well-designed. The retry/backoff configuration is thorough. The TenantAwareJob base class is solid. The notification dispatch processor with its fallback chain and per-row retry is production-quality. The architecture documentation's self-awareness of risks (36 documented danger zones) is itself a reliability feature.

**What prevents 8+**: Worker Sentry gap. Empty catch blocks. Stub worker health. Single-queue BullMQ monitoring. Escalation crash window.
