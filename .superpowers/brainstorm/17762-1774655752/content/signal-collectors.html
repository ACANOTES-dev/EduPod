Section 2: Signal Collectors

One thin adapter per domain. Queries existing tables, outputs a normalized SignalResult for the scoring engine.

Collector Interface

interface SignalResult {
  domain: 'attendance' | 'grades' | 'behaviour' | 'wellbeing' | 'engagement';
  rawScore: number; // 0-100 normalized
  signals: DetectedSignal[]; // individual data points
  summaryFragments: string[]; // NL sentences for this domain
}

interface DetectedSignal {
  signalType: string; // e.g. 'consecutive_absences'
  severity: 'low' | 'medium' | 'high' | 'critical';
  scoreContribution: number; // how many points this adds
  details: Record<string, unknown>; // signal-specific data
  sourceEntityType: string;
  sourceEntityId: string;
  summaryFragment: string; // "Absent 3 consecutive days (Mar 25-27)"
}

AttendanceSignalCollector

attendance-signal.collector.ts
DATA SOURCES
DailyAttendanceSummary (last 30 days)  ·  AttendancePatternAlert (active alerts)  ·  AttendanceRecord (recent for streak calc)
SIGNALS DETECTED
attendance_rate_decline 30-day attendance % below tenant threshold (default 90%) +10-30 pts
consecutive_absences 3+ consecutive days absent (excl. school closures) +15-25 pts
recurring_day_pattern Same day absent 3+ times in 4 weeks (mirrors existing alert) +10-20 pts
chronic_tardiness Late arrivals > 20% of attended days in 30-day window +5-15 pts
attendance_trajectory Week-over-week attendance rate declining for 3+ consecutive weeks +10-20 pts
SCORING
rawScore = min(100, sum(signal contributions)). Capped at 100. Leverages existing AttendancePatternAlert records — if an active alert exists, its type maps directly to a signal.

rawScore = min(100, sum(signal contributions)). Directly consumes existing StudentAcademicRiskAlert records — at_risk_low/medium/high map to score bands. Adds missing assessment and trajectory signals on top.

behaviour-signal.collector.ts
DATA SOURCES

incident_frequency 3+ negative incidents in 14-day window +10-25 pts

active_sanction Currently serving detention/suspension +15-30 pts
exclusion_history Any exclusion case this academic year +20-35 pts

SCORING
rawScore = min(100, sum(signal contributions)). Consumes existing BehaviourAlert records for pattern-based signals. Adds sanction/exclusion/intervention signals. Only counts negative polarity incidents.

EngagementSignalCollector

engagement-signal.collector.ts
DATA SOURCES
Notification (parent-targeted, last 30 days)  ·  ParentInquiry (activity this year)  ·  StudentParent → Parent → User (last_login_at)  ·  BehaviourParentAcknowledgement (response times)
SIGNALS DETECTED
low_notification_read_rate Less than 30% of in-app notifications read in last 30 days +10-20 pts
no_portal_login No parent has logged in within configurable days (default 21) +15-25 pts
no_parent_inquiry Zero parent-initiated communication this academic year +5-15 pts
slow_acknowledgement Behaviour incident acknowledgement takes >72 hours average +10-20 pts
disengagement_trajectory Login frequency and read rate both declining over 4 weeks +10-20 pts
SCORING
rawScore = min(100, sum(signal contributions)). Resolves student → parents via StudentParent junction. If student has multiple parents, uses the most engaged parent's metrics (best-case, not worst-case) to avoid penalising single-parent households.

Architecture Notes

Each collector is a standalone injectable class — @Injectable() with PrismaService dependency only. No cross-collector dependencies. The scoring engine calls all 5 and merges results.
Collectors reuse existing alert records — AttendancePatternAlert, StudentAcademicRiskAlert, BehaviourAlert are consumed as inputs, not duplicated. The early warning system adds cross-domain correlation on top.
Score ranges are configurable per signal type — the min/max point ranges shown above are defaults stored in early_warning_configs.weights_json. Schools can tune individual signal contributions.
DZ-27 respected — WellbeingSignalCollector never touches survey_responses (no tenant_id, anonymity-by-design). Only uses student-attributed pastoral records.
Engagement uses best-parent metric — for fairness. A student with one very engaged parent and one disengaged parent shouldn't be flagged for engagement risk. This is a deliberate design choice to avoid false positives in separated families.
