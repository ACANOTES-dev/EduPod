Section 3: Scoring Engine

Pure computation — no DB access. Takes SignalResults in, produces RiskAssessment out. The reusable core for ETB.

Scoring Pipeline

Computation Flow
1
Receive 5× SignalResult
attendance.rawScore=65  ·  grades.rawScore=72  ·  behaviour.rawScore=40  ·  wellbeing.rawScore=55  ·  engagement.rawScore=80
↓
2
Apply tenant weights
Default: attendance=25%, grades=25%, behaviour=20%, wellbeing=20%, engagement=10%
weighted = (65×0.25) + (72×0.25) + (40×0.20) + (55×0.20) + (80×0.10) = 61.25
↓
3
Cross-domain correlation boost
If 3+ domains have rawScore ≥ 40: apply +5 boost (multi-signal concern)
If 4+ domains have rawScore ≥ 40: apply +10 boost (systemic concern)
If all 5 domains have rawScore ≥ 40: apply +15 boost (critical pattern)
4 domains ≥ 40 → +10 boost → 71.25
↓
4
Assign tier (with hysteresis)
See hysteresis logic below
71.25 ≥ amber threshold (50) → tier = amber
↓
5
Generate NL summary + trend
Merge summaryFragments from all collectors. Sort by score contribution (highest first). Prepend trend sentence.
"Risk score increased from 48 to 71 over the past 3 weeks. Maths grade dropped from B+ to C- this period. Absent 4 of last 10 school days. No parent portal login in 26 days. Two behaviour incidents in the last 14 days."
↓
6
Return RiskAssessment
{ compositeScore, riskTier, domainScores, signals, summaryText, trendData, tierChanged, previousTier }

Hysteresis Logic

Tier Transition Rules — Prevents Alert Fatigue
GREEN 0-29
YELLOW 30-49
AMBER 50-74
RED 75-100
Upgrading (getting worse) — immediate:
Score crosses threshold → tier changes immediately
Score 71 → 75: instantly promoted to RED
Rationale: deterioration should never be delayed
Downgrading (improving) — requires buffer:
Score must drop hysteresis_buffer (default 10) points below current tier's entry threshold
Student in RED (entered at 75): must score ≤ 65 to drop to AMBER
Student in AMBER (entered at 50): must score ≤ 40 to drop to YELLOW
Student in YELLOW (entered at 30): must score ≤ 20 to drop to GREEN
Rationale: prevents oscillation at boundaries, ensures genuine improvement before removing alerts
Example — no oscillation:
Day 1: score 73 → AMBER
Day 2: score 76 → RED (upgrade, immediate)
Day 3: score 72 → stays RED (72 > 65 = threshold - buffer)
Day 4: score 68 → stays RED (68 > 65)
Day 5: score 64 → AMBER (64 ≤ 65, genuine improvement confirmed)
Without hysteresis, this student would have bounced RED→AMBER→RED generating 3 notifications

Cross-Domain Correlation Boost

This is the feature's core value — the thing no competitor does. A student with a 40 in attendance alone is mildly concerning. A student with 40 in attendance AND 40 in grades AND 40 in behaviour is a crisis pattern.

The boost is deliberately modest (+5/+10/+15) — it's a nudge, not a multiplier. It ensures multi-domain students surface above single-domain students with the same weighted average, without dramatically inflating scores.

The 40-point domain threshold for "concerning" is configurable per tenant in early_warning_configs.

Natural Language Summary

Template-based string construction. No AI API call. Fully deterministic. Each signal's summaryFragment is a pre-formatted sentence generated by its collector using concrete data.
// Scoring engine assembles the final summary:
const trendSentence = buildTrendSentence(currentScore, trendData);
// → "Risk score increased from 48 to 71 over the past 3 weeks."

const signalSentences = allSignals
  .sort((a, b) => b.scoreContribution - a.scoreContribution)
  .slice(0, 5) // top 5 contributors
  .map(s => s.summaryFragment);
// → ["Maths grade dropped from B+ to C-...", "Absent 4 of last 10...", ...]

return [trendSentence, ...signalSentences].join(' ');
EXAMPLE OUTPUT
"Risk score increased from 48 to 71 over the past 3 weeks. Maths grade dropped from B+ to C- this period. Absent 4 of the last 10 school days including 3 consecutive days (Mar 25-27). Two negative behaviour incidents in the last 14 days. No parent portal login in 26 days."

Output: RiskAssessment Interface

interface RiskAssessment {
  compositeScore: number; // 0-100
  riskTier: 'green' | 'yellow' | 'amber' | 'red';
  domainScores: {
    attendance: number;
    grades: number;
    behaviour: number;
    wellbeing: number;
    engagement: number;
  };
  crossDomainBoost: number; // 0, 5, 10, or 15
  signals: DetectedSignal[]; // all signals from all collectors
  summaryText: string; // the NL summary
  trendData: number[]; // last 30 daily composite scores
  tierChanged: boolean; // did the tier change this computation?
  previousTier: RiskTier | null; // null if first computation
}
